# 安全模型

## 一、鉴权方案 (ZAP)

### 核心思路：ZMQ 原生 PLAIN + ZAP 认证

```
1. 客户端连接时填写 ZMQ_PLAIN_USERNAME=api_key, ZMQ_PLAIN_PASSWORD=""
2. ZMQ 框架在连接握手阶段自动触发 ZAP 请求
3. 服务端 ZAP Handler 验证 api_key → 授权通过/拒绝
4. 授权通过后，服务端内存写入 identity → user_info 映射
5. 后续消息：服务端从 ZMQ identity 直接查内存，客户端无需每次携带凭证
6. ZMQ 内置心跳检测掉线 → 自动清理内存
```

#### 为什么不用应用层 AUTH 消息

| 对比 | 应用层 AUTH 消息（旧） | ZMQ PLAIN + ZAP（新） |
|---|---|---|
| 认证时机 | 连接后第一条业务消息 | 连接握手阶段 |
| 未认证窗口 | 存在（连接已建立但未 AUTH） | 不存在（握手不通过直接断开） |
| 安全性 | 应用层实现，易遗漏边界条件 | ZMQ 内核保证，RFC 27/28 标准 |
| 开发成本 | 需要实现 AUTH 状态机、超时、重试 | ZMQ 内置，只需实现 ZAP Handler |

---

### 连接协议选择

**PLAIN（非 TLS）**：内网场景，ZMQ TCP 内部传输，明文 api_key 可接受。

**CURVE（推荐）**：公网或安全意识要求高的场景，ZMQ CURVE 提供椭圆曲线加密 + 认证。

```
PLAIN 模式（内网）:
  客户端: ZMQ_PLAIN_USERNAME = api_key
  服务端: ZMQ_PLAIN_SERVER = 1

CURVE 模式（公网/生产）:
  客户端: ZMQ_CURVE_SERVERKEY = <server_public>
          ZMQ_CURVE_PUBLICKEY = <client_public>
          ZMQ_CURVE_SECRETKEY = <client_secret>
          ZMQ_PLAIN_USERNAME = api_key          ← CURVE 加密后传输
  服务端: ZMQ_CURVE_SERVER = 1
          ZMQ_CURVE_SECRETKEY = <server_secret>
          ZMQ_PLAIN_SERVER = 1                  ← 仍需 PLAIN 传递 api_key
```

两种模式切换仅需配置，代码无需改动。

---

### ZAP Handler 实现

```python
class PulseMQZAPHandler:
    """ZMQ ZAP (ZeroMQ Authentication Protocol) Handler

    ZMQ 在连接握手时自动调用，不需要应用层介入。
    """

    def __call__(self, domain, address, mechanism, username, password, client_key):
        # mechanism: "PLAIN" 或 "CURVE"
        # username:  客户端填的 api_key
        # password:  空字符串（不使用）

        user_info = self._validate_api_key(username)  # 查 DB
        if user_info is None:
            return self._reject("Invalid API key")

        if user_info.disabled:
            return self._reject("Account disabled")

        if self.connection_count(user_info.user_id) >= user_info.max_connections:
            return self._reject("Too many connections")

        # 认证通过 → 写入内存
        identity = address.encode()  # ZMQ 提供的客户端标识
        self.auth_store.register(identity, user_info)

        # 返回 user_id 作为 ZAP metadata（可选，供后续查询）
        return self._accept(user_id=str(user_info.user_id))
```

**关键**：ZAP Handler 运行在 ZMQ 的 IO 线程中，不阻塞 asyncio event loop。

### 连接建立全流程

```
Client                                        Server(ROUTER)
  │                                              │
  │── TCP 三次握手 ──────────────────────────►  │
  │                                              │
  │── ZMQ 握手 (PLAIN/CURVE + ZAP) ────────►   │
  │   username = api_key                         ├─ ZAP Handler 验证 api_key
  │                                              ├─ 通过 → 内存: identity → user_info
  │                                              ├─ 拒绝 → 断开连接
  │◄── 握手完成 ────────────────────────────│
  │                                              │
  │── SUB (topic) ─────────────────────────►   │
  │                                              ├─ 从 identity 查内存: O(1)
  │                                              ├─ 检查权限
  │                                              ├─ 注册订阅
  │◄── SUB_SUCCESS ─────────────────────────│
  │                                              │
  │── PUB (topic, payload) ────────────────►   │
  │                                              ├─ 从 identity 查内存: O(1)
  │                                              ├─ 检查 pub 权限
  │                                              ├─ 路由到订阅者
  │                                              │
```

**与旧方案的关键区别**：不再需要客户端发送 AUTH 消息。连接握手结束 = 已认证。未认证的连接根本不会建立。

### 内存鉴权存储结构

```
identity → user_info      映射 (bytes → AuthUser)
user_id → identities      集合 (int → set of bytes)，支持多连接
identity → last_heartbeat  心跳时间 (bytes → float)
```

**已移除**：`auth_token → identity` 反向映射（不再需要应用层 token）。

### AuthUser 结构

```python
@dataclass
class AuthUser:
    user_id: int
    role: str              # "admin" | "user"
    groups: List[str]      # 权限组名称列表
    api_key: str           # 原始 api_key（仅用于日志/审计，不做鉴权查询）
    namespace: str         # home_namespace
    max_connections: int   # 最大连接数
    disabled: bool
```

### 权限校验

采用**权限组（RBAC）方案**。

#### 流程

```
1. 从 ZMQ identity 获取用户 (O(1))
2. 检查用户角色
   - Admin → 全权限，直接通过
   - User → 进入步骤 3
3. 查用户展开权限缓存 (TTL=60s)
   缓存 key: user_perms:{user_id}
   {
     pub:  ["team-a.mkt.*", ...],
     sub:  ["*.mkt.*", ...]
   }
   命中 → 步骤 4
   未命中 → 查 DB (UserGroup + GroupPermission)
     → 按 action 分组展开为 topic_pattern 列表
     → 回写缓存
4. 匹配 topic_pattern（通配符 * 和 > 支持）
5. 权限变更时主动 invalidate 用户缓存
```

#### 缓存策略

- 缓存 key: `user_perms:{user_id}` — 展开后的权限列表
- TTL: 60 秒
- 失效时机：
  - 用户加入/离开 Group → 失效该用户缓存
  - Group 权限变更 → 失效该 Group 下所有成员缓存
  - 定时清理过期条目

---

### 用户角色

| 角色 | 权限 | 说明 |
|---|---|---|
| admin | 全权限 (MQ + 管理 API) | 跳过权限校验 |
| user | 按权限组授权 | 需校验加入的权限组的 topic_pattern + action |

### API Key 管理

| 属性 | 值 |
|---|---|
| 格式 | `pulse_sk_` + 32 位随机 hex（如 `pulse_sk_a1b2c3d4...`） |
| 存储 | DB 中 SHA-256 哈希存储，不存明文 |
| 生命周期 | 创建/启用/禁用/删除 |
| 轮换 | 一个用户可拥有多个 api_key，逐个轮换 |

#### API Key 操作接口（管理 API，非 ZMQ 消息）

```
POST /api/v1/admin/keys              # 创建 API Key → 返回明文（仅此一次）
GET  /api/v1/admin/keys              # 列出 Key（不返回明文）
POST /api/v1/admin/keys/{id}/disable # 禁用
POST /api/v1/admin/keys/{id}/enable  # 启用
DELETE /api/v1/admin/keys/{id}       # 删除
```

---

### 重连处理

客户端断开后重连：

```
1. 客户端重新 TCP + ZMQ 握手，再次传递 api_key
2. ZAP Handler 验证 api_key → 通过
3. 服务端清理旧 identity 的订阅（EVENT_DISCONNECTED 触发）
4. 绑定新 identity → user_info 映射
5. 客户端重新 SUB 需要的 topic
```

不需要 token 续期、不需要重连携带 token。每次重连就是一次新的完整握手。

## 二、权限架构 (RBAC)

### 核心决策

- 抛弃 namespace 概念，采用**权限组（RBAC）**
- 权限组 = 一组 topic_pattern + action 的集合
- 用户加入一个或多个权限组
- 用户最终权限 = 所有加入的权限组的**权限并集**

### 为什么不用 Namespace

- 你的场景不是多租户隔离（团队之间需要互相订阅行情）
- 成员的权限模式高度重复（同一团队的人权限基本一样）
- Namespace 概念多一层间接，管理成本高
- 权限组更直观：一目了然"这个人属于哪些组"

### 为什么不用 ACL（用户直连 Topic）

- 你的 topic 按 `team.symbol` 组织，数量可能上百
- 为每个用户在 100 个 topic 上逐条配置是不可接受的人力成本
- 权限组天然支持批量管理

---

### 数据模型

#### PermissionGroup（权限组）

```
id:         int
name:       str         "team-a-发布者"
created_at: float
```

#### GroupPermission（组与权限的关联）

```
id:             int
group_id:       int
topic_pattern:  str         "*.mkt.*"     ← 支持通配符 * 和 >
action:         str         "pub" | "sub" | "query"

UNIQUE(group_id, topic_pattern, action)
```

#### UserGroup（用户与组的关联）

```
id:         int
user_id:    int
group_id:   int

UNIQUE(user_id, group_id)
```

---

### 通配符规则

| 符号 | 含义 | 示例 | 匹配 | 不匹配 |
|---|---|---|---|---|
| `*` | 单层匹配 | `a.*.c` | `a.b.c` | `a.b.x.c`, `a.c` |
| `>` | 多层匹配 | `a.>.c` | `a.b.c`, `a.b.x.c`, `a.x.y.z.c` | `b.c`, `a.c` |

**示例**：

```
pattern: "*.mkt.*"
匹配: "team-a.mkt.sh.600000"
匹配: "team-b.mkt.sz.000333"
匹配: "team-c.mkt.bj.000001"
不匹配: "team-a.signal.buy"      ← 中间不是 mkt
不匹配: "team-a.mkt"             ← 少了第三层
```

---

### 典型配置示例

```
Group: 行情全订阅
  name: "行情全订阅"
  permissions:
    - *.mkt.*, sub

Group: 交易信号
  name: "交易信号"
  permissions:
    - *.signal.*, sub

Group: team-a-发布者
  name: "team-a-发布者"
  permissions:
    - team-a.mkt.*, sub
    - team-a.mkt.*, pub

Group: team-b-发布者
  name: "team-b-发布者"
  permissions:
    - team-b.mkt.*, sub
    - team-b.mkt.*, pub
```

**用户分配**：

```
user_1 (team-a 成员):
  加入: 行情全订阅 + 交易信号 + team-a-发布者
  最终权限:
    sub:  *.mkt.*, *.signal.*, team-a.mkt.*
    pub:  team-a.mkt.*

user_2 (只读行情):
  加入: 行情全订阅
  最终权限:
    sub:  *.mkt.*

user_3 (team-b 成员):
  加入: 行情全订阅 + team-b-发布者
  最终权限:
    sub:  *.mkt.*, team-b.mkt.*
    pub:  team-b.mkt.*
```

---

### Topic 命名规则

权限组方案下，topic**仍然可以包含组织前缀**（可视为标签，非权限层概念）：

```
Topic 格式:  "{org}.{category}.{code}"

示例:
  team-a.mkt.sh.600000     格力电器行情
  team-a.mkt.sz.000333     美的集团行情
  team-a.signal.buy        买入信号
  team-b.mkt.sh.600000     团队B的格力行情
  shared.market.index      公共市场指数
```

权限控制完全通过 GroupPermission 的 `topic_pattern` 匹配，不需要显式的 namespace 概念。

---

### 权限校验流程

```
PUB topic="team-a.mkt.sh.600000"

1. identity → user (O(1)，内存缓存)

2. user.role == "admin" → 直接通过，跳过后续

3. 获取用户展开后的权限（查缓存）
   缓存 key: user_perms:{user_id}
   {
     pub:  ["team-a.mkt.*"],
     sub:  ["*.mkt.*", "*.signal.*"]
   }
 
   缓存命中 → 步骤4
   缓存未命中 → 查 DB（表关联: user → user_group → group_permission）
     → 按 action 分组，展开为 topic_pattern 列表
     → 回写缓存（TTL 60s）

4. 匹配 topic
   "team-a.mkt.sh.600000" 匹配 pub 列表:
     - "team-a.mkt.*" → 匹配 ✓

5. 通过 → 继续消息处理
```

---

### 缓存策略

#### 用户权限展开缓存

```
key:       user_perms:{user_id}
value:
  {
    pub:  ["team-a.mkt.*", "team-a.signal.*", ...],
    sub:  ["*.mkt.*", "*.signal.*", ...],
    query: [...]
  }
TTL:       60 秒
来源:      UserGroup + GroupPermission 表关联查询后展开
```

#### 失效时机

| 事件 | 失效范围 |
|---|---|
| 用户加入/离开某个 Group | 仅该用户: `user_perms:{user_id}` |
| Group 的名称变更 | 不需要失效（鉴权不需要 Group 名称） |
| Group 的权限新增/删除 | 该 Group 下**所有成员**的用户缓存 |
| Group 被删除 | 该 Group 下**所有成员**的用户缓存（同时删除 UserGroup 关联） |
| 新增用户 | 创建时计算并写入缓存 |

---

### 权限变更流程

#### 新增用户 → 加入权限组

```
1. 插入 UserGroup(user_id, group_id)
2. 失效缓存: user_perms:{user_id}
3. 下次消息自动重算
```

#### 调整 Group 权限（面向团队）

```
1. 新增/删除 GroupPermission(group_id, topic_pattern, action)
2. 从 DB 查该 Group 的所有成员 → 逐个失效 user_perms:{user_id}
3. 所有成员下次消息自动重算
```

#### 用户退出某个 Team

```
1. 删除该用户此 Team 相关的 UserGroup 关联
2. 失效缓存: user_perms:{user_id}
```

---

### 与鉴权方案的结合

```
整体权限校验流程:

1. Transport.recv() → identity

2. AUTH 阶段:
   - 查 AuthMemoryStore: identity → user_info
   - user_info 中存: user_id, role, [permission_groups]
   - AUTH 阶段不查权限详情，只查身份

3. Pipeline 拦截器链 (每条消息):
   ┌─ AuthInterceptor: identity → user (O(1))
   ├─ PermInterceptor: user → 展开权限 → 匹配 topic × action
   └─ MonitorInterceptor: 记录指标
```

**性能关键**：
- `user.role == "admin"` 在 PermInterceptor 的第一个判断，直接短路跳过
- 用户展开权限缓存在内存中 (60s TTL)，不需要每条消息查 DB
- 通配符匹配在缓存中预编译为 pattern 列表，匹配是纯字符串操作

### 管理接口设计（V2 扩展）

V1 阶段，admin 用户**直接操作 SQLite** 完成权限组和用户管理：

```sql
-- 创建权限组
INSERT INTO permission_groups (name, created_at) VALUES ('team-a-发布者', unixepoch());
-- 添加权限规则
INSERT INTO group_permissions (group_id, topic_pattern, action) VALUES (1, 'team-a.mkt.*', 'pub');
-- 用户加入权限组
INSERT INTO user_groups (user_id, group_id) VALUES (2, 1);
```

V2 将通过 QUERY 协议暴露 REST 风格管理 API。届时支持以下操作（草案）：

| 操作 | QUERY action | 说明 |
|------|-------------|------|
| 列出权限组 | `list_groups` | 含 member_count |
| 创建权限组 | `create_group` | — |
| 删除权限组 | `delete_group` | 需先移除所有成员 |
| 权限组详情 | `get_group` | 含 permissions + members |
| 新增权限规则 | `add_permission` | topic_pattern + action |
| 移除权限规则 | `remove_permission` | — |
| 添加成员 | `add_member` | user_id → group_id |
| 移除成员 | `remove_member` | — |

### 设计优势总结

| 维度 | 说明 |
|---|---|
| 管理效率 | 新增成员只需加入几个 Group，批量调整权限只需改 Group |
| 灵活性 | 跨团队共享只需创建一个"共享行情"Group，有权限的人都加入 |
| 审计可读性 | 一目了然：用户属于哪些 Group → Group 有哪些权限 |
| 性能 | 权限展开后缓存到内存，60s TTL，O(1) 查缓存 + O(n) pattern 匹配 |
| 扩展性 | 新增角色模式 → 新建一个 Group 即可，不修改代码 |
