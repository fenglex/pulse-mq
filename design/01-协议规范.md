# 协议规范

## 一、消息帧格式

### 固定 6 帧格式

```
Frame 0: identity          [ZMQ ROUTER 自动附加，应用层不可见]
Frame 1: delimiter         [ZMQ 空帧分隔符，路由信封结束标志]
Frame 2: topic             [路由键，无 topic 时为空字节]
Frame 3: meta (2 bytes)    [msg_type(1B) + flags(1B)]
Frame 4: record_count      [4B big-endian uint32，固定存在]
Frame 5: payload           [序列化 + 压缩后的消息体]
```

**固定 6 帧，无条件**。不存在可变帧，编解码逻辑无需判断帧数，直接按索引读取。

### 两条路径展开

#### 控制路径 (DEALER → ROUTER)

```
客户端组装:   [topic][meta(2B)][record_count(4B)][payload]
ZMQ 自动附加: [identity][delimiter][topic][meta(2B)][record_count(4B)][payload]
服务端收到:   topic 从 Frame 2 直接读取
              record_count 从 Frame 4 直接读取（不解码，4B → uint32）
```

#### 数据路径 (XPUB → SUB)

```
服务端发送:   [topic][meta(2B)][record_count(4B)][payload]
              └── ZMQ 在这一帧做订阅匹配 ──┘
客户端收到:   只有订阅前缀匹配的才会到达应用层
```

### meta 帧编码

#### Byte 0: msg_type

| 值 | 类型 | 方向 | 说明 |
|---|---|---|---|
| 0x01 | AUTH | S→C | 连接成功后推送用户元信息 |
| 0x02 | PUB | C→S | 发布消息 |
| 0x03 | SUB | C→S, S→C | 订阅/订阅确认 |
| 0x04 | UNSUB | C→S, S→C | 取消订阅/确认 |
| 0x05 | QUERY | C→S, S→C | 管理查询/响应 |
| 0x06 | PING | C→S | 心跳请求 |
| 0x07 | PONG | S→C | 心跳响应 |
| 0x08 | STATUS | C→S, S→C | 状态上报/广播 |
| 0x09 | ERROR | S→C | 错误响应 |
| 0x0A | BROADCAST | S→C | 广播消息（XPUB→SUB 数据路径） |
| 0x0B | HISTORY_REPLAY | C→S, S→C | 请求/回放历史消息 |
| 0x0C-FF | reserved | - | 预留 |

#### Byte 1: flags bitfield

| 位 | 含义 | 编码 |
|---|---|---|
| bit[0:2] | 序列化格式 | 000=msgpack(default), 001=raw, 010=pyarrow, 011=protobuf, 100-111=reserved |
| bit[3:4] | 压缩算法 | 00=none(default), 01=snappy, 10=lz4, 11=zstd |
| bit[5] | has_topic | 0=无topic, 1=有topic |
| bit[6] | reserved | 预留 |
| bit[7] | reserved | 预留 |

**压缩算法编码**：

| bit[3:4] | 压缩算法 | 说明 |
|---|---|---|
| 00 | none | 不压缩（默认） |
| 01 | snappy | 极速压缩，Google 出品 |
| 10 | lz4 | 极速压缩/解压，比 snappy 更快 |
| 11 | zstd | Facebook Zstandard，高压缩比 + 可调速度 |

4 种算法覆盖全部场景：零压缩 / 极速 / 高压缩比。gzip/zlib/brotli 等按需自行注册。

### 各消息类型帧布局

所有消息统一 6 帧。record_count 语义因消息类型不同：

| 类型 | Frame 2 (topic) | Frame 3 (meta) | Frame 4 (record_count) | Frame 5 (payload) |
|---|---|---|---|---|
| AUTH | 空字节 | 0x01 + has_topic=0 | 0（控制消息） | 用户元信息 |
| PUB | `mkt.sh.600000` | 0x02 + has_topic=1 + ser+comp | ≥1（数据行数） | 序列化消息体 |
| SUB | `mkt.sh.*` | 0x03 + has_topic=1 | 0（控制消息） | 空或订阅选项 |
| UNSUB | `mkt.sh.600000` | 0x04 + has_topic=1 | 0（控制消息） | 空 |
| PING | 空字节 | 0x06 + has_topic=0 | 0（控制消息） | 客户端时间戳 |
| PONG | 空字节 | 0x07 + has_topic=0 | 0（控制消息） | 时间戳 + server_time |
| QUERY | 可变 | 0x05 | 0（控制消息） | 查询参数 |
| STATUS | 空或topic | 0x08 | 0（控制消息） | 状态数据 |
| ERROR | 可变 | 0x09 | 0（控制消息） | error_code + message |
| BROADCAST | `mkt.sh.600000` | 0x0A + has_topic=1 + ser+comp | ≥1（数据行数） | 序列化消息体 |
| HISTORY_REPLAY | `mkt.sh.600000` | 0x0B + has_topic=1 + ser+comp | ≥1（数据行数） | 序列化消息体 |

#### record_count 语义

```
record_count = 0  → 控制消息，payload 不含数据行，忽略此字段
record_count ≥ 1  → 数据消息，payload 含 N 行数据（单条消息 N=1）
```

### 与旧格式对比

| 维度 | 旧格式 (6帧) | 新格式 (6帧) |
|---|---|---|
| 帧数 | 6 | 6（固定，不区分路径） |
| record_count | 不支持 | 固定 Frame 4，4B uint32 |
| topic 位置 | 藏在 payload 里 | 独立帧（Frame 2），O(1) 读取 |
| ZMQ 订阅过滤 | 不支持 | 原生支持（内核级过滤） |
| 序列化格式 | 单独一帧 "msgpack" (7B) | 装在 flags 的 3 bits |
| 压缩算法 | 单独一帧 "gzip" (4B) | 装在 flags 的 2 bits |
| 扩展性 | 加字段需加帧 | flags 剩余 3 bits 可用 |
| 引擎解码次数 | 每条消息都要解码 payload | 路由路径零解码 |

---

### record_count 设计说明

#### 为什么需要独立帧

PulseMQ 面向金融行情场景，一条 PUB 消息可能携带**批量数据**（如 pyarrow DataFrame 的 500 行 tick 数据）。如果不区分"消息条数"和"数据行数"，会产生以下问题：

| 问题 | 没有 record_count | 有 record_count |
|---|---|---|
| 吞吐量指标 | `msg_rate = 100` 含义模糊（100 条消息？100 行数据？） | `record_rate = 50,000` 准确反映实际数据量 |
| MessageBuffer 容量 | 存 1000 条"消息"可能实际存了 500,000 行 → 内存爆炸 | 按消息条数存，每条的 record_count 只是 uint32 |
| 序列号 _seq | 批量是 1 条还是 N 条？客户端无法判断是否丢数据 | _seq 按消息递增，record_count 报告消息内数据量 |
| 计费/配额 | 无法按实际数据量限流 | 可以按 record_count 做精细的速率控制 |

#### 为什么固定帧而非可变帧

| 对比 | 可变帧 (bit[7] 标记) | 固定帧 (始终 6 帧) |
|---|---|---|
| 编解码逻辑 | 需判断 bit[7] 决定帧数，分支逻辑 | 固定索引读取，零分支 |
| 帧解析 | 条件读取 record_count | 无条件读取，uint32 直接转 |
| 误判风险 | bit[7] 解析错误 → 帧偏移错位 → 后续全部乱套 | 不会错位 |
| 帧数差异 | 大部分消息 5 帧，少量 6 帧 | 统一 6 帧，4B 开销可忽略 |
| ZMQ 多帧性能 | 少一帧略快 | 多一帧可忽略（ZMQ 合并发送） |

**结论**：4 字节的额外开销可以忽略不计，但固定帧带来的确定性远优于可变帧。调试、抓包分析、边界检查都更简单。

#### 编码方式

```
4 字节 big-endian uint32
控制消息: 0x00 0x00 0x00 0x00
单条数据: 0x00 0x00 0x00 0x01
500 行数据: 0x00 0x00 0x01 0xF4
```

#### 与 _seq 的关系

```
_seq     = 消息序号（每次 PUB 调用 +1，不受 record_count 影响）
record_count = 本次 PUB 携带的数据行数（≥1）

示例:
  PUB #1: {"price": 15.8}                  → _seq=1, record_count=1
  PUB #2: DataFrame[500 rows]               → _seq=2, record_count=500
  PUB #3: {"price": 15.9}                  → _seq=3, record_count=1

客户端可以通过累加 record_count 计算实际收到的数据行数:
  total_rows = sum(msg.record_count for msg in received if msg.record_count > 0)
```

#### 引擎层处理

固定帧无需判断，直接按索引读取：

```
引擎 recv 6 帧:
  frames[2] → topic（字节）
  frames[3] → meta（2B）
  frames[4] → record_count（4B → int）
  frames[5] → payload（字节，零拷贝转发）
  
  if record_count > 0:
    写入监控指标（msg_rate=1, record_rate=record_count）
  payload 原样传递给 handler
```

### 核心优势

#### 1. ZMQ 内核级订阅过滤

- ZMQ SUB socket 通过 `setsockopt(SUBSCRIBE, prefix)` 做前缀匹配
- 匹配的是 PUB 发送的**第一帧的开头字节**
- 匹配成功 → 收全量帧；不匹配 → 内核层直接丢弃，应用层无感知
- 对于行情分发场景 (100 symbol, 每人订阅 5 个)，**95% 的消息在内核层丢弃**，不唤醒 asyncio 事件循环

#### 2. 引擎层零解码

- topic 从 Frame 2 直接读取字节
- msg_type 从 Frame 3[0] 单字节偏移
- flags 从 Frame 3[1] 单字节偏移
- record_count 从 Frame 4 直接 4 字节转 uint32
- 权限校验直接用 topic 查缓存
- **payload 始终不被解码**（除非特殊消息类型需要）

#### 3. 零拷贝转发

- PUB Handler 权限校验通过后
- 直接转发 `frames[2:]`（topic + meta + record_count + payload）给 XPUB
- 不经过反序列化 → 重序列化流程
- 引擎层零解码（topic、meta、record_count 都是裸字节读取）

#### 4. 固定帧确定性的优势

- 无分支逻辑 → 编解码路径单一，易于测试和验证
- 帧偏移固定 → 抓包分析时直接按索引定位，不需要回溯 bit[7]
- 6 帧恒定 → ZMQ send/recv 调用次数固定，性能可预测
- 4 字节开销 → 对于动辄 KB 级别的行情 payload，可忽略不计

## 二、消息类型完整定义

所有消息类型统一使用固定 6 帧格式。帧布局详见第一节，msg_type 枚举定义见第一节 meta 帧编码中的 msg_type 表。差异在于 Frame 3 的 flags 和 Frame 5 的 payload 内容。

### 1. AUTH（用户元信息推送）

认证本身由 ZMQ PLAIN/ZAP 在连接握手阶段完成。AUTH 消息仅用于连接建立后，服务端向客户端推送用户元信息。

```
服务端 → 客户端（ZMQ 握手成功后自动发送）:
  meta:         0x01 + has_topic=0
  topic:        空字节
  record_count: 0x00000000         ← 控制消息，固定为 0
  payload (msgpack):
    {
      "user_id": 1,
      "role": "user",
      "namespace": "team-a",
      "groups": ["行情全订阅", "team-a-发布者"],
      "server_time": 1717516800.123
    }
```

不再有客户端→服务端的 AUTH 请求。认证在 ZMQ 层完成，客户端拿到的是纯元信息。不需要 auth_token。不需要 error 响应（ZMQ 握手失败直接断开连接）。

### 2. PUB（发布消息）

```
客户端 → 服务端:

  单条消息:
    meta:         0x02 + has_topic=1 + ser_fmt + comp
    topic:        "team-a.mkt.sh.600000"
    record_count: 0x00000001  ← 固定 4B，值=1
    payload:      序列化+压缩后的消息体（单条数据）

  批量消息（DataFrame 等）:
    meta:         0x02 + has_topic=1 + ser_fmt + comp
    topic:        "team-a.mkt.sh.600000"
    record_count: 0x000001F4  ← 500 行
    payload:      序列化+压缩后的消息体（500 行数据）

服务端 → 客户端:
  不发直接响应（PUB 是 fire-and-forget）
  如果权限校验失败 → 发 ERROR 消息
```

### 3. SUB / UNSUB（订阅/取消订阅）

```
客户端 → 服务端:
  meta:         0x03 + has_topic=1
  topic:        "team-a.mkt.*"
  record_count: 0x00000000         ← 控制消息，固定为 0
  payload (msgpack, 可选):
    {
      "options": {
        "durable": false,     ← 预留
        "max_rate": null      ← 预留
      }
    }

服务端 → 客户端:
  成功:
    meta:         0x03 + has_topic=1
    topic:        "team-a.mkt.*"
    record_count: 0x00000000         ← 控制消息，固定为 0
    payload (msgpack):
    {
      "status": "ok",
      "expanded_topics": [
        "team-a.mkt.sh.600000",
        "team-a.mkt.sz.000333"
      ]
    }

  失败:
    meta:         0x09
    topic:        "team-a.mkt.*"
    record_count: 0x00000000         ← 控制消息，固定为 0
    payload:
    {
      "code": 2001,
      "message": "Permission denied: cannot subscribe to team-a.mkt.*"
    }
```

UNSUB 同理，msg_type=0x04，payload 结构相同。

### 4. PING / PONG（心跳）

```
客户端 → 服务端:
  meta:         0x06 + has_topic=0
  topic:        空字节
  record_count: 0x00000000         ← 控制消息，固定为 0
  payload (msgpack):
    {
      "client_ts": 1717516800.123
    }

服务端 → 客户端:
  meta:         0x07 + has_topic=0
  topic:        空字节
  record_count: 0x00000000         ← 控制消息，固定为 0
  payload (msgpack):
    {
      "client_ts": 1717516800.123,
      "server_ts": 1717516800.456
    }
```

**使用说明**：ZMQ 内置心跳覆盖掉线检测（HEARTBEAT_IVL=2s, TIMEOUT=5s），应用层 PING/PONG 仅用于**端到端延迟采样**和**应用层活性探测**，不需高频发送。

### 5. QUERY（管理查询）

#### 查询系统状态（V1 支持）

```
请求:
  meta:         0x05 + has_topic=0
  topic:        空字节
  record_count: 0x00000000
  payload:
    {"action": "system_status"}

响应:
  meta:         0x05
  topic:        空字节
  record_count: 0x00000000
  payload:
    {
      "status": "ok",
      "timestamp": 1717516800.456,
      "uptime_seconds": 3600,
      "topic_count": 128,
      "subscription_count": 1024,
      "connection_count": 42,
      "msg_rate": 15342.5,
      "record_rate": 245000.0,
      "bytes_rate": 1048576
    }
```

#### 管理操作（V2 扩展）

V1 阶段，以下管理操作**不通过 QUERY 协议暴露**。admin 直接操作 SQLite 完成用户/权限组管理：

```
# V1 直接操作数据库
sqlite3 pulse_mq.db

-- 创建权限组
INSERT INTO permission_groups (name, created_at) VALUES ('行情全订阅', unixepoch());
-- 添加用户
INSERT INTO users (username, api_key, role, namespace, created_at, updated_at)
VALUES ('user_a', 'pulse_sk_xxx', 'user', 'team-a', unixepoch(), unixepoch());
-- 用户加入权限组
INSERT INTO user_groups (user_id, group_id) VALUES (2, 1);
```

V2 将通过 QUERY 协议暴露 REST 风格的管理 API（`list_groups`、`create_group`、`add_permission`、`add_member` 等）。

### 6. STATUS（状态上报，预留）

```
客户端 → 服务端（端到端延迟采样）:
  meta:         0x08 + has_topic=1
  topic:        "team-a.mkt.sh.600000"
  record_count: 0x00000000         ← 控制消息，固定为 0
  payload (msgpack):
    {
      "type": "e2e_latency",
      "receive_ts": 1717516800.789
    }
```

服务端不返回响应，仅用于监控指标采集。

### 7. ERROR（服务端 → 客户端）

```
meta:         0x09 + has_topic=0/1  ← 根据上下文中是否有 topic
topic:        原来的 topic（如果有）
record_count: 0x00000000            ← 控制消息，固定为 0
payload (msgpack):
  {
    "code": 2001,
    "message": "Invalid API key",
    "details": {}
  }
```

### 8. BROADCAST（服务端 XPUB → 客户端 SUB）

```
meta:         0x0A + has_topic=1 + ser_fmt + comp
topic:        "team-a.mkt.sh.600000"
record_count: 固定 4B uint32（原样转发，服务端不修改）
payload:      序列化+压缩后的消息体（原样转发，零拷贝）

与 PUB 的区别:
  PUB = 客户端发布给服务端 (DEALER→ROUTER)
  BROADCAST = 服务端广播给订阅者 (XPUB→SUB)
  客户端收到后根据 ser_fmt+comp 解码 payload
  所有帧全部原样转发，服务端不修改任何字段

序列号（用于消息回放）:
  每条 PUB 消息在 Broker 端分配 topic 内单调递增的序列号
  _seq 嵌入 payload 的 msgpack dict 中，不占用 meta 帧空间
  客户端解码后可获取:
    {
      "_seq": 1523,           ← Broker 分配的消息序号（每次 PUB +1）
      "_ts": 1717516800.456,  ← Broker 接收时间戳
      ... 原始数据字段 ...
    }
```

### 9. HISTORY_REPLAY（历史消息回放）

```
客户端 → 服务端（请求回放）:
  meta:         0x0B + has_topic=1
  topic:        "team-a.mkt.sh.600000"
  record_count: 0x00000000         ← 控制消息，固定为 0
  payload (msgpack):
    {
      "from_seq": 1500,      ← 从哪个序列号开始（0=从最早可用）
      "limit": 100           ← 最多回放多少条（默认 100，最大 500）
    }

服务端 → 客户端（回放消息）:
  逐条发送 BROADCAST 消息（从 MessageBuffer 取出）:
    meta:         0x0A + has_topic=1 + 原始 ser_fmt + comp
    topic:        "team-a.mkt.sh.600000"
    record_count: 原始值（原样恢复）
    payload:      原始序列化+压缩后的消息体（含 _seq, _ts 字段）

服务端 → 客户端（回放结束）:
    meta:         0x0B + has_topic=1
    topic:        "team-a.mkt.sh.600000"
    record_count: 0x00000000         ← 控制消息，固定为 0
    payload (msgpack):
    {
      "status": "ok",
      "from_seq": 1500,
      "to_seq": 1599,
      "count": 100,
      "latest_seq": 1600     ← 当前 topic 最新序列号（客户端可判断是否追上）
    }

失败:
    meta:         0x09
    topic:        "team-a.mkt.sh.600000"
    record_count: 0x00000000         ← 控制消息，固定为 0
    payload:
    {
      "code": 2001,          ← 权限不足
      "message": "Permission denied: cannot read topic"
    }
```

## 三、错误码

### 结构

```
ERROR 消息 payload:
{
  "code": 2001,
  "message": "Permission denied: cannot subscribe to team-a.mkt.*",
  "details": {}                    ← 可选，额外诊断信息
}
```

---

### 完整错误码

#### 1000-1999: 认证

| 错误码 | 消息 | 触发条件 |
|---|---|---|
| 1001 | Invalid API key | api_key 不存在或格式错误（ZAP 握手阶段） |
| 1002 | Account disabled | 账号被禁用 |
| 1003 | Too many connections | 同一用户连接数超限 |
| 1004 | Auth required | 未经认证就发送非 AUTH 消息 |

#### 2000-2999: 权限

| 错误码 | 消息 | 触发条件 |
|---|---|---|
| 2001 | Permission denied | 用户没有该 topic+action 的权限 |
| 2002 | Topic not found | topic 不存在且不允许自动创建 |
| 2003 | Rate limit exceeded | 用户级别 QPS 超限 |
| 2004 | Not an admin | 需要 admin 角色的操作 |
| 2005 | Cannot kick self | 不能踢掉自己 |

#### 3000-3999: 管理操作

| 错误码 | 消息 | 触发条件 |
|---|---|---|
| 3001 | Invalid parameters | 参数缺失或格式错误 |
| 3002 | Resource already exists | 创建的 Group/User 已存在 |
| 3003 | Resource not found | 查询/操作的 Group/User 不存在 |
| 3004 | Invalid action | action 字段不合法（不是 pub/sub/query） |
| 3005 | Invalid topic pattern | topic_pattern 通配符语法错误 |
| 3006 | Group name empty | 权限组名称为空 |
| 3007 | Username empty | 用户名为空 |
| 3008 | Cannot modify admin | 不能修改 admin 用户的权限组 |

## 四、序列化格式与压缩算法注册表

### 设计模式

注册表模式：运行时注册新算法，引擎和客户端共享同一个注册表实例。

---

### 序列化格式注册表

#### 抽象接口

```python
class Serializer(ABC):
    @abstractmethod
    def serialize(obj: Any) -> bytes: ...
    @abstractmethod
    def deserialize(data: bytes) -> Any: ...

class SerializationRegistry:
    @classmethod
    def register(name: str, serializer: Serializer): ...
    @classmethod
    def get(name: str) -> Serializer: ...
    @classmethod
    def list() -> List[str]: ...
    @classmethod
    def has(name: str) -> bool: ...
```

#### 内置格式

##### raw（纯字节流 — 所有格式的基础层）

```
注册名: "raw"
flags:  bit[0:2] = 001
说明:   不做任何序列化，直接透传 bytes。这是 PulseMQ 消息传输的字节基础层
        —— 所有其他序列化格式（msgpack/pyarrow/protobuf）最终都产出 bytes，
        通过同一套帧格式传输。raw 就是最原始形态的"已序列化完成的 bytes"。
序列化: obj (必须为 bytes) → return obj
反序列化: data → return data
适用:   纯二进制数据、已由业务层自行序列化的数据
性能:   零开销（裸指针透传）
```

##### msgpack（默认）

```
注册名: "msgpack"
flags:  bit[0:2] = 000
说明:   二进制 JSON，紧凑且快速
序列化: dict/list → msgpack.packb
反序列化: bytes → msgpack.unpackb
适用:   通用场景，大多数消息
性能:   ~10-50μs (1KB payload)
库:     msgpack-python
```

##### pyarrow

```
注册名: "pyarrow"
flags:  bit[0:2] = 010
说明:   Apache Arrow 列式格式，零拷贝
序列化: Table/RecordBatch → pa.ipc.serialize() → bytes
反序列化: bytes → pa.ipc.deserialize() → Table/RecordBatch
适用:   行情结构化数据 (大数组、DataFrame)
性能:   列式压缩，适合批量行情数据
库:     pyarrow >= 10.0（pa.serialize 已废弃，使用 pa.ipc 模块）
```

##### protobuf

```
注册名: "protobuf"
flags:  bit[0:2] = 011
说明:   Google Protocol Buffers，强类型跨语言序列化
注册方式: 运行时注册（唯一方式）——用户实现 Serializer 接口并注册
序列化: message_instance → .SerializeToString()
反序列化: bytes + MessageClass → .ParseFromString(data)
适用:   强类型接口，跨语言兼容
性能:   ~5-20μs (1KB payload)
库:     protobuf
```

**使用方式**：用户注册自定义 ProtobufSerializer，绑定自己的 MessageClass：

```python
from pulse_mq.serialization import registry, Serializer
from my_proto import MarketData

class MarketDataSerializer(Serializer):
    def serialize(self, obj: MarketData) -> bytes:
        return obj.SerializeToString()
    def deserialize(self, data: bytes) -> MarketData:
        msg = MarketData()
        msg.ParseFromString(data)
        return msg

registry.register("marketdata", MarketDataSerializer())
client.publish("topic", msg, format="marketdata")
```

#### 可扩展格式（预留）

以下 flags 槽位为用户自行注册使用，非内置：

| 格式 | flags | 说明 |
|---|---|---|
| json | 100 | 文本 JSON（调试用，不推荐生产） |
| flatbuffers | 101 | 零拷贝序列化 |
| capnproto | 110 | 零拷贝序列化 |

扩展注册示例见上文 protobuf 章节。

### 压缩算法注册表

#### 抽象接口

```python
class Compressor(ABC):
    @abstractmethod
    def compress(data: bytes) -> bytes: ...
    @abstractmethod
    def decompress(data: bytes) -> bytes: ...

class CompressionRegistry:
    @classmethod
    def register(name: str, compressor: Compressor): ...
    @classmethod
    def get(name: str) -> Compressor: ...
    @classmethod
    def list() -> List[str]: ...
    @classmethod
    def has(name: str) -> bool: ...
```

#### 内置算法

##### none（默认，不压缩）

```
注册名: "none"
flags:  bit[3:4] = 00
说明:   不压缩，直接透传
压缩:   data → return data
解压:   data → return data
适用:   小消息、低延迟场景
性能:   零开销
```

##### snappy

```
注册名: "snappy"
flags:  bit[3:4] = 01
说明:   Google Snappy，极致速度
压缩:   data → snappy.compress(data)
解压:   data → snappy.decompress(data)
适用:   高频行情数据，需要极低延迟
性能:   ~5μs (1KB), 压缩比 ~1.5:1
库:     python-snappy
```

##### lz4

```
注册名: "lz4"
flags:  bit[3:4] = 10
说明:   LZ4，极速压缩/解压，比 snappy 更快
压缩:   data → lz4.frame.compress(data)
解压:   data → lz4.frame.decompress(data)
适用:   极致速度要求
性能:   ~3μs (1KB), 压缩比 ~1.3:1
库:     lz4
```

##### zstd

```
注册名: "zstd"
flags:  bit[3:4] = 11
说明:   Facebook Zstandard，平衡压缩比和速度
压缩:   data → zstd.compress(data, level=3)
解压:   data → zstd.decompress(data)
适用:   需要高压缩比且保持低延迟
性能:   ~20μs (1KB), 压缩比 ~4:1
库:     zstandard
```

#### 压缩算法 flags 汇总

| bit[3:4] | 算法 | 注册名 | 内置 |
|---|---|---|---|
| 00 | none | none | ✅ |
| 01 | snappy | snappy | ✅ |
| 10 | lz4 | lz4 | ✅ |
| 11 | zstd | zstd | ✅ |

4 种内置算法覆盖全部场景。其他算法（gzip/zlib/brotli 等）通过注册表按需注册。

---

### 序列化 + 压缩联合使用

#### 处理顺序

```
发布端:  payload → serialize → compress → 网络
接收端:  网络 → decompress → deserialize → payload
```

#### 完整处理函数

```
def encode_payload(obj, serializer_name, compressor_name) -> bytes:
    serializer = SerializationRegistry.get(serializer_name)
    compressor = CompressionRegistry.get(compressor_name)
    return compressor.compress(serializer.serialize(obj))

def decode_payload(data, serializer_name, compressor_name) -> Any:
    serializer = SerializationRegistry.get(serializer_name)
    compressor = CompressionRegistry.get(compressor_name)
    return serializer.deserialize(compressor.decompress(data))
```

#### 帧中编码

```
Frame 3: meta (2B)
  Byte 0: msg_type
  Byte 1: flags
    bit[0:2] = 序列化格式 (000=msgpack, 001=raw, 010=pyarrow, 011=protobuf)
    bit[3:4] = 压缩算法 (00=none, 01=snappy, 10=lz4, 11=zstd)
    bit[5]   = has_topic
    bit[6:7] = reserved

客户端解析: 直接读 bit[3:4] 获取压缩算法，无需扩展位判断
```

---

### 算法选择建议

| 场景 | 序列化 | 压缩 | 理由 |
|---|---|---|---|
| 通用 | msgpack | none | 平衡，默认 |
| 高频行情 (tick) | pyarrow | snappy | 列式 + 极速压缩 |
| 纯二进制 / 业务自行序列化 | raw | none | 零开销透传 |
| 大数组行情 | pyarrow | zstd | 列式 + 高压缩比 |
| 跨语言强类型 | protobuf | none | 强类型约束 |
| 极低延迟数据 | raw | lz4 | 最快压缩/解压 |
| 批量历史数据 | pyarrow | zstd | 最高效的列式压缩 |

内置 4 种压缩算法已覆盖所有场景，其余算法按需注册。
