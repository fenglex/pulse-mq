## 一、监控设计

### 整体架构

```
┌─────────────────────────────────────┐
│         Realtime (内存)              │
│  EWMA + Counter + SlidingWindow     │
│  精度: 秒级                         │
│  API: GET /api/v1/metrics/realtime  │
├─────────────────────────────────────┤
│         Persistent (SQLite)         │
│  分钟聚合槽 → 异步批量写入           │
│  保留: 7天自动清理                   │
│  独立 stats.sqlite，不影响业务DB     │
│  历史查询: 通过 Prometheus endpoint  │
│            导出后外部查询（V1 不做   │
│            内置历史查询 API）        │
└─────────────────────────────────────┘
```

### 实时指标

#### 内存结构

```
RealtimeMetrics:
  msg_rate:          EWMA(α=0.3)          消息帧速率（每条 PUB 计数 1）
  record_rate:       EWMA(α=0.3)          数据行速率（累计 record_count）
  bytes_rate:        EWMA(α=0.3)          字节速率
  latency_p50:       SlidingWindow(60s)   处理延迟 P50
  latency_p99:       SlidingWindow(60s)   处理延迟 P99
  active_connections:  int                当前连接数
  active_subscriptions: int               当前订阅数
  error_rate:        EWMA(α=0.3)          错误速率
  topic_msg_rate:    Dict[str,EWMA]       各 topic 消息帧速率
  topic_record_rate: Dict[str,EWMA]       各 topic 数据行速率
  dropped_messages:  Counter              背压丢弃消息计数
  backpressure:      bool                 是否处于背压状态
```

### 分钟指标

#### SQLite Schema

```sql
-- stats.sqlite（独立文件，与业务DB隔离）

CREATE TABLE minute_topic_stats (
    ts          INTEGER NOT NULL,    -- unix timestamp (分钟精度)
    topic       TEXT NOT NULL,
    msg_count   INTEGER DEFAULT 0,
    bytes_total INTEGER DEFAULT 0,
    latency_p50 REAL DEFAULT 0,
    latency_p99 REAL DEFAULT 0,
    error_count INTEGER DEFAULT 0,
    PRIMARY KEY (ts, topic)
);
CREATE INDEX idx_minute_topic_ts ON minute_topic_stats(ts);

CREATE TABLE minute_user_stats (
    ts          INTEGER NOT NULL,
    user_id     INTEGER NOT NULL,
    pub_count   INTEGER DEFAULT 0,
    sub_count   INTEGER DEFAULT 0,
    error_count INTEGER DEFAULT 0,
    PRIMARY KEY (ts, user_id)
);
CREATE INDEX idx_minute_user_ts ON minute_user_stats(ts);

CREATE TABLE minute_system_stats (
    ts              INTEGER NOT NULL PRIMARY KEY,
    total_msgs      INTEGER DEFAULT 0,
    total_bytes     INTEGER DEFAULT 0,
    avg_latency_ms  REAL DEFAULT 0,
    peak_connections INTEGER DEFAULT 0,
    peak_subscriptions INTEGER DEFAULT 0,
    error_total     INTEGER DEFAULT 0,
    dropped_total   INTEGER DEFAULT 0
);
```

#### MySQL 迁移兼容性

Schema 设计确保未来零改动迁移：

| 设计决策 | 现状 | MySQL 兼容 |
|---|---|---|
| ts 用整数 (unix ts) | ✅ | `FROM_UNIXTIME(ts)` 转换即可 |
| 联合主键 (ts, topic) | ✅ | MySQL 同样支持 |
| 索引在 ts 上 | ✅ | 分区键候选 |
| 无自增 ID | ✅ | 联合主键更高效 |
| 类型对标 | INTEGER/REAL/TEXT | INT/DOUBLE/VARCHAR |

迁移步骤：
1. 创建表（SQL 兼容）
2. 实现 MysqlMetricsRepo（实现同一个 ABC）
3. 数据迁移脚本

### 分钟数据写入

#### 流程

```
每分钟定时任务:
  1. 切换槽位:
     完成槽 = current_slot
     current_slot = MinuteSlot()
     槽开始时间 = now

  2. 异步写入 SQLite:
     - 批量 INSERT 三条
     - 独立 DB 连接 + 独立 SQLite 文件
     - 不阻塞主循环

  3. 每小时清理:
     DELETE FROM minute_*_stats WHERE ts < (now - 7*86400)
```

#### 性能保障

| 保障 | 实现 |
|---|---|
| 实时指标零阻塞 | inc_*() 无锁，event loop 单线程内调用，O(1) |
| 分钟写入零影响 | 独立 asyncio task + 独立 DB 连接 + 独立 SQLite 文件 |
| 查询零干扰 | 实时读内存，历史由外部工具负责 |

### 二、单元测试

#### 2.1 帧编解码

| 测试项 | 覆盖内容 | 预期 |
|---|---|---|
| 基本编解码 | 标准 6 帧消息的 encode/decode 往返 | 完全一致 |
| meta flags 位操作 | has_topic=1, ser_fmt=msgpack(000), comp=none(00) | 正确解析各 bit 位 |
| 序列化格式切换 | msgpack / raw / pyarrow / protobuf 四种格式 | 各格式编码正确 |
| 压缩算法切换 | none / snappy / lz4 / zstd 四种算法 | 压缩→解压往返一致 |
| 序列化+压缩组合 | msgpack+zstd, pyarrow+snappy 等典型组合 | 编码→解码正确 |
| 边界条件 | 空 payload、极大 payload (1MB)、空 topic、identity 含特殊字节 | 不崩溃，正确编解码 |
| 畸形帧 | 不足 6 帧、meta 字节数错误、未知 ser_fmt | 抛出 DecodeError |
| 注册表扩展 | 运行时注册新 Serializer/Compressor | 注册后 encode/decode 可用 |

```python
# 示例测试
@pytest.mark.parametrize("ser,comp", [
    ("msgpack", "none"),
    ("msgpack", "snappy"),
    ("pyarrow", "zstd"),
    ("bytes", "lz4"),
])
async def test_frame_roundtrip(ser, comp):
    original = {"price": 15.8, "volume": 10000}
    encoded = FrameCodec.encode("test.topic", original, ser, comp)
    topic, decoded = FrameCodec.decode(encoded)
    assert topic == "test.topic"
    assert decoded == original
```

#### 2.3 权限展开逻辑

| 测试项 | 覆盖内容 |
|---|---|
| 单 Group 展开 | user 属于 1 个 group → 正确展开为 pub/sub 列表 |

#### 2.4 ZAP Handler

| 测试项 | 覆盖内容 |
|---|---|
| 有效 api_key | ZAP 请求携带正确 api_key → ACCEPT |
| 无效 api_key | ZAP 请求携带随机字符串 → REJECT |
| 禁用账号 | api_key 对应的 user.disabled=True → REJECT |

#### 2.5 MessageBuffer 环形覆盖

| 测试项 | 覆盖内容 |
|---|---|
| 基本追加 | append N 条 → buffer 长度 = N |
| 环形覆盖 | append 1005 条（MAX_SIZE=1000）→ buffer 长度 = 1000，最旧的 5 条被淘汰 |
| 序列号递增 | 追加消息 → seq 单调递增（1,2,3,...） |
| 从指定 seq 回放 | append 100 条 → replay(from_seq=50) 返回 seq 50-100 |
| 从最早回放 | replay(from_seq=0) → 返回 buffer 中全部消息 |
| 空 topic 回放 | replay 不存在的 topic → 返回空列表 |
| limit 限制 | append 200 条 → replay(limit=50) → 返回 50 条 |
| topic 清理 | remove_topic → buffer 不存在，seq_counter 重置 |
| 多 topic 隔离 | 在 topic A 和 B 分别 append → 各自独立 seq 计数 |

#### 2.5.1 对象池 (MessagePool)

| 测试项 | 覆盖内容 |
|---|---|
| 基本获取归还 | acquire → 设置字段 → release → 再次 acquire → 字段已清空 |
| 池耗尽回退 | acquire 超过 pool_size 次 → 回退到普通分配，不崩溃 |
| 归还后无残留 | release 后再次 acquire 同一对象 → 所有字段为空/默认值 |
| 预分配计数 | 初始化 pool_size=100 → pool 预分配 100 个 Message 对象 |
| 并发安全 | 多个协程并发 acquire/release → 无竞态（单线程 event loop） |
| Message.reset() | release 调用 reset → identity/topic/meta/payload 全部清空 |

### 四、混沌测试

#### 4.1 背压触发与恢复

```
测试: 高负载触发背压 → Producer 降速 → 背压解除

1. 启动 服务端，设置 max_concurrency=10
2. 启动 50 个 Producer，每个以 1000 msg/s 持续发送
3. 启动 1 个慢 Consumer（处理每条消息 sleep 100ms）
4. 观察:
   - 信号量使用率上升到 80% → 服务端 暂停 recv
   - Producer 端 send 开始超时/失败
   - Consumer 逐渐消费积压 → 信号量下降
   - 服务端 恢复 recv → Producer 恢复
5. 验证: 无消息丢失（全部消息要么成功到达，要么 Producer 明确收到失败）
6. 验证: 无 OOM（内存在合理范围内波动）
```

#### 4.2 大量连接抖动

```
测试: 100 个客户端同时断开+重连

1. 启动 100 个客户端，全部认证+订阅
2. 同时杀掉所有客户端进程
3. 等待 5s (HEARTBEAT_TIMEOUT)
4. 验证: 服务端 内存中无残留订阅，ConnectionManager 为空
5. 所有客户端重新连接
6. 验证: 全部认证成功，订阅恢复
7. 验证: 服务端 在抖动期间未崩溃，CPU/内存正常
```

#### 4.3 信号量耗尽

```
测试: 信号量满时 服务端 行为

1. 设置 max_concurrency=5
2. 启动 20 个慢 Consumer（处理极慢）
3. Producer 持续发送
4. 验证:
   - 信号量满 → 服务端 暂停 recv（不崩溃）
   - ZMQ RCVHWM 触发 → socket 缓冲区满
   - Producer 感知背压（send 超时或 ZMQ 丢弃）
5. 杀掉所有慢 Consumer
6. 验证: 信号量释放 → 服务端 恢复 → 新连接正常工作
```

#### 4.4 畸形消息注入

```
测试: 发送各种异常消息，服务端 不崩溃

异常类型:
  - msg_type 非法值（0xFF）
  - 未认证连接直接 PUB
  - 帧数不足（只有 3 帧）
  - payload 不是合法 msgpack（乱码字节）
  - topic 包含非法字符（空格、null 字节）
  - meta flags 声明 msgpack 但 payload 是 protobuf

验证:
  - 服务端 不崩溃、不 panic
  - 对畸形消息返回 ERROR 或忽略
  - 其他正常连接不受影响
```

#### 4.5 消息缓冲区内存压测

```
测试: 极端 topic 数量下的 MessageBuffer 内存行为

1. 向 1000 个 topic 每个发送 10000 条消息（超过 MAX_SIZE 10 倍）
2. 验证: 每个 topic 只保留最新 1000 条
3. 验证: 内存占用稳定在 ~100MB（不持续增长）
4. 向 100 个 topic 持续发送 1 小时（每秒 100 条）
5. 验证: 内存无泄漏（1 小时后内存与 1 分钟时基本一致）
```

### 五、性能基准

#### 5.1 基准测试框架

```
工具: pytest-benchmark
环境: 同机器 localhost，排除网络延迟
消息: 1KB payload, msgpack + none 压缩
```

#### 5.2 基准指标

| 指标 | 目标（单 服务端，单 Producer） | 测量方法 |
|---|---|---|
| 吞吐量 | > 50,000 msg/s | 1 个 Producer 持续 PUB，统计 30s 内消息数 |
| P50 延迟 | < 500μs | Producer send → Consumer recv 往返时间 |
| P99 延迟 | < 5ms | 同上 |
| 内存占用 (空载) | < 50MB | 服务端 启动后 RSS（无连接、无消息） |
| 内存占用 (1000 topic × 1000 msg) | < 250MB | 满载 MessageBuffer 后 RSS（含 slots 优化） |
| CPU (空载) | < 1% | 无消息时 CPU 使用率 |
| CPU (满载) | 充分利用可用核心 | 50,000 msg/s 时 CPU 使用率 |
| 连接建立时间 | < 10ms | TCP + ZMQ 握手完成 |
| SUB 响应时间 | < 1ms | SUB 请求 → SUB_SUCCESS 时间 |

#### 5.2.1 uvloop vs 标准 asyncio 对比

```
测试: 同硬件、同消息负载下，服务端 分别使用 uvloop 和标准 asyncio

| 指标           | asyncio (标准) | uvloop        | 提升 |
|----------------|---------------|---------------|------|
| 吞吐量         | ~25000 msg/s  | ~50000 msg/s  | 2x   |
| P50 延迟       | ~800μs        | ~400μs        | 2x   |
| P99 延迟       | ~8ms          | ~4ms          | 2x   |
| 空载 CPU       | ~1%           | ~0.5%         | —    |
| 连接建立时间   | ~15ms         | ~10ms         | 1.5x |
```

uvloop 基线是 PulseMQ 默认性能目标。标准 asyncio 仅作为 Windows 回退方案。

#### 5.3 序列化格式对比

```
测试: 1KB dict 消息，测量 encode + decode 耗时

| 格式      | encode (μs) | decode (μs) | payload 大小 | 说明 |
|-----------|-------------|-------------|-------------|------|
| msgpack   | ~10         | ~8          | ~850B       | 默认 |
| raw       | ~0          | ~0          | 1024B       | 字节透传，零开销 |
| pyarrow   | ~40         | ~25         | ~700B       | pa.ipc 模块 (pyarrow ≥10.0) |
| protobuf  | ~8          | ~5          | ~800B       | 需预编译 .proto，运行时注册 |
```

#### 5.4 压缩算法对比

```
测试: 1KB msgpack 编码的行情 dict，测量压缩/解压耗时和压缩比

| 算法   | compress (μs) | decompress (μs) | 压缩比 | 适用 |
|--------|---------------|-----------------|--------|------|
| none   | ~0            | ~0              | 1:1    | 默认，零开销 |
| snappy | ~5            | ~3              | 1.5:1  | 高频行情，极低延迟 |
| lz4    | ~3            | ~2              | 1.3:1  | 极致速度 |
| zstd   | ~20           | ~10             | 4:1    | 高压缩比 + 可调速度 |

推荐: 高频行情用 snappy/lz4（极低延迟），批量数据用 zstd（高压缩比）。
gzip/zlib/brotli 等按需自行注册。
```

#### 5.5 并发扩展性

```
测试: N 个 Producer + N 个 Consumer，测量吞吐量

| Producer/Consumer 对 | 吞吐 (msg/s) | P99 延迟 |
|---------------------|-------------|----------|
| 1:1                 | 50,000      | 3ms      |
| 5:5                 | 200,000     | 5ms      |
| 10:10               | 350,000     | 8ms      |
| 20:5                | 180,000     | 12ms     | ← Consumer 瓶颈
```
