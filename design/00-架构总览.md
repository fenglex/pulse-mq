# 架构总览

## 一、层级架构

### 4 层架构

```
Layer 1 · 接入层 (Access Layer)
  定位: 对外暴露入口，协议编解码，客户端 SDK
  依赖: Layer 4 接口（Transport、Serialization）

Layer 2 · 引擎层 (Engine Layer)
  定位: 消息核心引擎，路由，拦截器链
  依赖: Layer 1（协议定义）、Layer 3（调用领域服务）

Layer 3 · 领域层 (Domain Layer)
  定位: 业务逻辑，认证授权，权限管理
  依赖: Layer 4（存储接口）

Layer 4 · 基础设施层 (Infrastructure Layer)
  定位: 数据持久化，缓存，监控，配置
  依赖: 无（最底层）
```

### 依赖规则

- 箭头只向下，上层依赖下层**接口**，不依赖实现
- 同层模块可互相调用
- 下层不知道上层的存在（事件通知除外）

### 各层功能

#### Layer 4 · 基础设施层

| 模块 | 功能 | 扩展性 |
|---|---|---|
| Transport | ZMQ 收发字节，管理 socket 生命周期 | 可替换为 TCP/gRPC |
| Serialization | raw/msgpack/pyarrow 序列化 | 可加入 protobuf/JSON |
| Compression | none/snappy/lz4/zstd 压缩 | 其余按需注册 |
| Storage | 元数据持久化 (User/Topic/Permission) | 可替换 MySQL/PostgreSQL/Redis |
| Cache | 内存缓存 (LRU + TTL) | 可接入 Redis 集群 |
| Monitoring | 指标采集 (实时 + 分钟级) | 可接入 Prometheus/StatsD |
| EventLoop | asyncio 事件循环引擎，默认使用 uvloop（Linux） | 自动回退到标准 asyncio |
| Config | 配置加载 (文件 + 环境变量) | — |

**关键约束**：
- Storage 只存元数据：User / PermissionGroup / PermissionRule
- **不存消息** (Message)、**不存订阅** (Subscription)、**不存连接** (Connection)
- 这些都是纯内存数据结构，重启丢失

#### Layer 3 · 领域层

| 模块 | 功能 |
|---|---|
| Models | User / Topic / Message / Namespace / Permission 实体定义（纯数据结构） |
| AuthService | API Key 认证（ZAP Handler） |
| PermissionService | 权限校验（用户 → 权限组 → topic_pattern → action） |
| TopicService | Topic 创建/查询/删除，自动注册 |

**关键设计**：
- Service 通过接口调用 Storage，不依赖具体实现
- 领域事件用于解耦横切关注点
- 用户角色只有 2 种：admin(全权限) + user(按权限组授权)

#### Layer 2 · 引擎层

| 模块 | 功能 |
|---|---|
| Engine | 消息主循环，批处理 + 并发控制 |
| Router | Topic 注册表 + 订阅管理 + 连接映射（全部内存） |
| Pipeline | 可插拔拦截器链（Auth → Perm → Monitor → ...） |
| Handler | 消息类型分发（AUTH/PUB/SUB/QUERY/PING/STATUS） |

**消息流**：

```
Client → Transport.recv()
       → Engine._message_loop()     [批处理 + 并发]
       → Pipeline.execute()         [拦截器链]
       → Handler.dispatch()         [消息分发]
       → Router.get_subscribers()   [O(1) 查找]
       → Transport.broadcast()      [XPUB 广播]
```

**扩展点**：
- 新拦截器实现 Interceptor 接口并加入链
- 未来集群可替换 Router 为分布式实现

#### Layer 1 · 接入层

| 模块 | 功能 |
|---|---|
| Protocol | 消息编解码 (6 帧格式)，序列化/压缩注册表 |
| Transport | ZMQ 适配 (适配到 Layer 4 Transport 接口) |
| Client SDK | 异步客户端 |
| Server | 启动器 (组装各层，启动服务) |

### EventLoop 优化：uvloop

#### 决策

使用 **uvloop** 替换 Python 标准 asyncio 事件循环，作为 Broker 的默认异步引擎。

#### 依据

| 维度 | 标准 asyncio | uvloop | 说明 |
|------|-------------|--------|------|
| 事件循环吞吐 | 基准 | **2-4x** | libuv (C) 实现，Node.js 同款引擎 |
| 连接建立延迟 | 基准 | **降低 20-30%** | 减少 Python 层对象分配 |
| CPU 占用 | 基准 | **略降低** | 更少的 Python 字节码执行 |
| 代码改动 | — | **1 行** | `uvloop.install()` |
| 兼容性 | 全平台 | **仅 Linux/macOS** | Windows 自动回退标准 asyncio |

#### 实现

```python
# pulsemq/engine/event_loop.py
import asyncio

def install_event_loop():
    """安装高性能事件循环，Linux 下使用 uvloop，否则回退标准 asyncio"""
    try:
        import uvloop
        uvloop.install()
        return "uvloop"
    except ImportError:
        return "asyncio"
```

#### Windows 兼容

Windows 不支持 uvloop（libuv IOCP 后端与 asyncio ProactorEventLoop 不兼容）。在 Windows 上自动回退到标准 asyncio，功能无损失，仅吞吐量略低。生产部署在 Linux 上不受影响。

#### 收益

对 PulseMQ 这种 IO 密集型 + 大量协程调度的场景，uvloop 减少每次事件循环迭代的 Python/C 边界穿越次数，在高负载下（pending_tasks > 50）效果尤为明显。

### 对象池优化：MessagePool

#### 决策

消息处理路径上的高频临时对象（`Message` 上下文、`BufferedMessage`）使用对象池复用，避免频繁 `malloc/free` 触发 GC。

#### 依据

| 维度 | 每次 new | 对象池复用 | 说明 |
|------|---------|-----------|------|
| 内存分配 | 每次 malloc | 仅首次预分配 | ~50-100x fewer allocations |
| GC 压力 | 高（短生命周期对象） | 极低 | 对象不进入 GC 各代 |
| CPU 开销 | PyObject_New 调用 | 指针归还 | 约 5-10x faster |
| 代码复杂度 | 简单 | 中等 | 需正确 reset() 状态 |

#### 适用对象

| 对象 | 生命周期 | 池策略 |
|------|---------|--------|
| `Message` (双缓冲条目) | recv → dispatch 完成 | 预分配 + acquire/release |
| `BufferedMessage` (MessageBuffer) | append → 环形淘汰 | 淘汰时回收进池 |
| Pipeline `context` dict | 单条消息处理周期 | 预分配 + reset 复用 |

#### 代价

- 对象池代码 ~30 行
- 需要确保 release 前无外部引用持有旧数据（无悬垂指针风险——Python 引用计数保证安全）
- 调试时对象状态可能含残留数据（reset 清空即可）

---

### 走内存 vs 走 Storage

| 走内存 (重启丢失) | 走 Storage (重启保留) |
|---|---|
| 消息本体 | 用户信息 |
| 订阅关系 | 权限规则 |
| 连接状态 | Topic 元数据 |
| Topic 注册表 | 命名空间 |
| 实时监控指标 | 分钟监控指标 |

## 二、传输协议选型

### 场景特征

- 行情数据实时分发（tick 级别，高频）
- 多订阅者（一个 topic 可能被几十个 consumer 订阅）
- Topic 过滤（按股票代码/品种过滤）
- 低延迟要求（金融场景 < 1ms 优先）
- 未来可能需要 Web 端接入

### 四种协议对比

#### 性能指标

| 维度 | ZMQ (ROUTER-DEALER) | 原生 TCP | WebSocket | gRPC |
|---|---|---|---|---|
| 单连接吞吐量 | ★★★★☆ | ★★★★★ | ★★☆☆☆ | ★★★☆☆ |
| 连接数扩展性 | ★★★★★ | ★★★☆☆ | ★★★★☆ | ★★★☆☆ |
| 延迟（单跳） | ~10-50μs | ~5-20μs | ~1-5ms | ~0.5-2ms |
| 吞吐量 64B 消息 | ~500K-1M/s | ~1-2M/s | ~50-100K/s | ~100-300K/s |
| 吞吐量 1KB 消息 | ~200-400K/s | ~500-800K/s | ~30-60K/s | ~80-150K/s |

#### 功能特性

| 维度 | ZMQ | 原生 TCP | WebSocket | gRPC |
|---|---|---|---|---|
| 消息帧定界 | ✅ 内置多帧 | ❌ 需自建 | ✅ Frame 协议 | ✅ Length-prefixed |
| 多路复用 | ✅ 内置 | ❌ 需自建 | ✅ 内置 | ✅ HTTP/2 |
| 自动重连 | ✅ 内置 | ❌ 需自建 | ✅ 协议层 | ✅ keepalive |
| 背压/流控 | ✅ HWM 机制 | ❌ 需自建 | ✅ TCP backpressure | ❌ 需自建 |
| 订阅过滤 | ✅ XPUB/SUB | ❌ 需自建 | ❌ 需应用层 | ❌ 需应用层 |
| 安全传输 | ✅ CURVE/ZAP | ✅ TLS | ✅ WSS (TLS) | ✅ TLS + Auth |
| 跨语言客户端 | ✅ 官方绑定 | 需自建 SDK | ✅ 浏览器原生 | ✅ 代码生成 |
| 浏览器直连 | ❌ | ❌ | ✅ | ❌ 需 grpc-web |
| 部署复杂度 | 中 (需 libzmq) | 低 (OS 内置) | 低 | 中 (protobuf) |

### 逐协议分析

#### ZMQ — 当前选择，推荐继续使用

**优势**：
- PUB/SUB、ROUTER/DEALER、REQ/REP 等消息模式开箱即用
- HWM 背压控制，防止内存爆炸
- XPUB 订阅过滤：服务端做 topic 过滤，不匹配的消息不发到网络
- 零拷贝 API (`zmq_msg_init_data()`)

**劣势**：
- 额外帧头开销
- 需要编译 libzmq（pyzmq prebuilt wheel 已解决）
- 协议私有，Wireshark 无法直接解析
- 连接模型固定

**适合场景**：行情数据分发、多订阅者、topic 过滤 —— 这些都是你的核心需求。

#### 原生 TCP — 性能天花板但开发成本高

**优势**：
- 绝对最低延迟，系统调用直达网卡
- 最高吞吐量，零额外封装
- 零依赖，OS 内置

**劣势**：
- 需要自建约 800-1000 行基础设施代码：
  - 帧协议设计（~200 行）
  - 连接管理器（~300 行）
  - 订阅过滤器（~150 行）
  - 背压控制（~100 行）
  - TLS 集成（~100 行）

**适合场景**：微秒级延迟要求（高频交易场景才值得）

#### WebSocket — 浏览器桥接

**优势**：
- 浏览器原生支持（WebTrader 场景）
- 穿透防火墙/NAT
- 天然全双工

**劣势**：
- 帧开销大（最小 2 字节头 + 4 字节 MASK key）
- 延迟高（HTTP Upgrade ~50-200ms）
- 无订阅过滤（所有消息发客户端，浪费带宽）
- 不适合高频数据

**适合场景**：Web 端轻量网关（WS → ZMQ bridge）

#### gRPC — 管理 API

**优势**：
- 强类型 IDL（.proto 定义接口）
- HTTP/2 多路复用
- 流式 RPC

**劣势**：
- 延迟偏高（protobuf 序列化 + HTTP/2 帧封装）
- 不适合 PUB/SUB（server streaming 是 1:1，不是 1:N 广播）
- 重量级（protobuf 编译工具链）

**适合场景**：管理 API（用户 CRUD、权限查询）

### 推荐架构

```
主协议: ZMQ (ROUTER-DEALER + XPUB-SUB)
辅助:   WebSocket (Web 端桥接), gRPC (管理 API)

协议分工：
├── Producer → Core: ZMQ DEALER-ROUTER   (数据发布)
├── Core → Consumer: ZMQ XPUB-SUB        (数据广播)
├── Browser → Core:   WebSocket Bridge   (Web 展示)
└── Admin → Core:     gRPC/HTTP REST    (管理操作)
```

### 性能优化优先级（投入产出比从高到低）

| 优先级 | 优化项 | 预计提升 |
|---|---|---|
| 1 | 消息循环并发化 | 5-10x |
| 2 | 编码缓存 + 零拷贝 | 2-3x |
| 3 | 监控异步隔离 | 减少抖动 |
| 4 | ZMQ socket 参数调优 | 10-30% |
| 5 | 替换为原生 TCP | 2x (但成本极高) |

### 结论

**继续用 ZMQ 作为主协议，不建议替换。**
- ZMQ 在行情数据分发场景中已经是最优解之一
- TCP 能带来 2-3x 吞吐提升，但要付出 800+ 行基础设施代码成本
- WebSocket 和 gRPC 作为辅助协议，特定场景使用
- 真正影响性能的是消息循环并发化，不是传输协议

## 三、配置管理

### 设计原则

- **零配置启动**：不提供任何配置时，PulseMQ 以合理默认值启动（内存队列 + SQLite）
- **代码启动**：支持在代码中编程式创建和启动服务
- **命令行启动**：`pulse-mq --config server.toml` 或 `pulse-mq --port 5555`
- **环境变量**：所有配置项可通过环境变量覆盖

### 零配置启动

```bash
# 零配置启动（全部默认值）
pulse-mq

# 等价于:
#   pulse-mq --transport zmq --bind tcp://*:5555
#            --db sqlite://./pulse_mq.db --stats-db sqlite://./stats.sqlite
#            --concurrency 100 --hwm 10000
```

```python
# 代码零配置启动
from pulse_mq import PulseServer

server = PulseServer()
await server.start()
```

### 配置来源优先级

```
命令行参数 > 配置文件 > 环境变量 > 默认值
```

### 配置文件格式

```toml
# pulse_mq.toml

[transport]
bind = "tcp://*:5555"
xpub_bind = "tcp://*:5556"

[database]
url = "sqlite://./pulse_mq.db"
stats_url = "sqlite://./stats.sqlite"
stats_retention_days = 7

[engine]
max_concurrency = 100
max_batch_size = 64
drain_timeout_ms = 1
use_uvloop = true
object_pool_size = 4096

[zmq]
rcvhwm = 10000
sndhwm = 10000
heartbeat_ivl = 2000
heartbeat_timeout = 5000
heartbeat_ttl = 8000

[overload]
data_buffer_size = 9000
ctrl_buffer_size = 1000
backpressure_threshold = 0.8

[metrics]
bind = "0.0.0.0:9090"
enabled = true

[auth]
enabled = true
default_admin_key = "pulse_sk_admin_default"
```

### 命令行启动

```bash
# 零配置
pulse-mq

# 指定端口
pulse-mq --bind tcp://*:5555 --xpub-bind tcp://*:5556

# 从配置文件
pulse-mq --config pulse_mq.toml

# 混合: 配置文件 + 命令行覆盖
pulse-mq --config pulse_mq.toml --concurrency 200

# 环境变量覆盖
PULSEMQ_CONCURRENCY=200 pulse-mq

# 禁用监控
pulse-mq --no-metrics
```
