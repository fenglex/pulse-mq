# 引擎设计

## 一、消息主循环并发模型

### 选型决策

采用**方案 D：自适应批处理 + 流水线派发**。经对比淘汰：

| 淘汰方案 | 致命缺陷 |
|---|---|
| A: 简单并发（fire-and-forget） | 每条消息单独 create_task，高负载时开销累积 |
| B: 同步批处理（gather 等整批） | 尾延迟增加，吞吐量受限 |
| C: 流水线（缺自适应/背压） | 没有 pending_tasks 背压，高负载可能内存泄漏 |

### 事件循环基础

并发模型运行在 **uvloop**（Linux/macOS）或标准 asyncio（Windows 回退）之上。uvloop 基于 libuv (C 实现)，提供 2-4x 事件循环吞吐提升。

---

### 核心设计

#### 三条独立路径解耦

```
主循环:    recv → 快速排空 → 按 topic 分组 → 派发 task（不等待完成）
工作协程:  信号量控制并发 → pipeline → handler → 完成后释放
条件唤醒:  同一 topic 的事务按序串行，不同 topic 的事务完全并发
```

主循环只负责"最快把消息从 socket 取出来"，不等待处理完成。

#### 同 Topic 有序保证（核心约束）

**约束**：同一 topic 的消息必须按到达顺序处理，不允许并发乱序。

**实现**：按 topic 分组派发，每个 topic 的消息交给同一个 task 顺序处理。

```
batch (收了一批消息):
  msg_A: topic=team-a.mkt.sh.600000  (到达顺序: 1)
  msg_B: topic=team-a.signal.alert   (到达顺序: 2)
  msg_C: topic=team-a.mkt.sh.600000  (到达顺序: 3)
  msg_D: topic=team-b.mkt.sh.600000  (到达顺序: 4)

分组后:
  topic "team-a.mkt.sh.600000"  → [msg_A, msg_C]  ← 同一个 task 顺序处理
  topic "team-a.signal.alert"   → [msg_B]          ← 独立 task
  topic "team-b.mkt.sh.600000"  → [msg_D]          ← 独立 task

→ 3 个 task 并发执行，但 topic 内严格保序
```

**为什么不在 服务端 端做全序**：全序意味着所有消息串行，吞吐量直接坍塌。同 topic 有序是金融场景的最小必要约束（同一只股票的数据不能乱序），不同 topic 之间无需顺序保证。

#### 完整流程

```
while running:
  1. 检查背压
     if pending_tasks > max_concurrency * 0.8:
       等待 pending_tasks 降到阈值以下

  2. recv 第一条（阻塞等待）
     decode: identity, topic, meta, payload
     加入 batch

  3. 排空缓冲区
     最多再收 effective_batch_size - 1 条
     每条 1ms 超时

  4. 按 topic 分组派发
     grouped = group_by_topic(batch)
     for topic, msgs in grouped.items():
       pending_tasks++
       task = create_task(_process_topic_batch(topic, msgs))
       task完成后 → pending_tasks--

_process_topic_batch(topic, msgs):
  async with sem(并发上限=100):
    for msg in msgs:           ← topic 内顺序处理，保证有序
      try:
        pipeline.execute(context)
        handler.dispatch(context)
      except ...:
        ...

  5. 更新自适应批大小
     effective_batch_size = adapt(batch_size_history)

_process_topic_batch(topic, msgs):
  """同一 topic 的消息在单个 task 内顺序处理，保证 topic 内有序"""
  async with sem(并发上限=100):
    for msg in msgs:               # ← 串行循环，保证顺序
      try:
        pipeline.execute(context)      # 拦截器链
        handler.dispatch(context)       # 消息分发
      except AuthError:
        send_error(identity, "AUTH_FAILED")
      except PermissionError:
        send_error(identity, "PERM_DENIED")
      except Exception:
        log + 监控
```

---

### 配置参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| `max_concurrency` | 100 | 信号量上限，同时处理的最大消息数 |
| `initial_batch_size` | 1 | 初始批大小（低负载时每条即时处理） |
| `max_batch_size` | 64 | 最大批大小（高负载时批量收） |
| `drain_timeout_ms` | 1 | 排空缓冲区时的单条超时 |
| `backpressure_threshold` | 0.8 | 信号量使用率到 80% 即暂停 recv |
| `adapt_window` | 10 | 自适应窗口大小（最近 N 次观察） |

---

### 自适应批大小

#### 调整逻辑

```
高负载判定: 最近 adapt_window 次中，连续排空（实际收到 >= effective_batch_size * 0.8）
  → effective_batch_size = min(effective_batch_size * 2, max_batch_size)

低负载判定: 最近 adapt_window 次中，连续收不满（实际收到 < 2 条）
  → effective_batch_size = max(effective_batch_size / 2, 1)
```

#### 效果

| 负载状态 | effective_batch_size | 延迟特征 |
|---|---|---|
| 低负载 | 1 | 每条消息即时处理，微秒级延迟 |
| 中负载 | 2→8→16 | 渐进式批量，平衡吞吐和延迟 |
| 高负载 | 64（上限） | 最大吞吐，尾延迟可控（仍受信号量保护） |

---

### 背压传导链路

```
Producer 发送过快
    ↓
Server 信号量满 (pending_tasks > 80% 阈值)
    ↓
Server 暂停 recv（停止从 socket 取消息）
    ↓
ZMQ ROUTER socket 内部缓冲区填满 → RCVHWM 触发
    ↓
ZMQ 丢弃新消息或阻塞 Producer 的 send
    ↓
Producer 感知失败/超时 → 降低发送速率或告警
```

**两条防线**：

| 防线 | 位置 | 机制 |
|---|---|---|
| 应用层 | pending_tasks 阈值 | 80% 信号量使用率即暂停 recv |
| 内核层 | ZMQ RCVHWM | socket 缓冲区满 → 拒绝/阻塞 |

**关键**：整个链路是**无锁、无额外组件**的纯协程背压，靠着信号量阈值和 ZMQ HWM 两条防线。

---

### 错误处理

#### 处理策略

```
每条消息独立 try/catch:

AuthError → 返回 AUTH_FAILED，断开连接
PermissionError → 返回 PERM_DENIED，继续服务
DecodeError → 返回 BAD_MESSAGE，继续服务
其他异常 → log + 监控 + 返回 INTERNAL_ERROR，继续服务
```

**核心理念**：单条消息失败不影响其他消息。一条 bad message 不杀死整个连接。

#### 并发安全

```
信号量保护:
  - 每条消息处理独占一个信号量槽
  - 处理完成后自动释放
  - context 对象每条消息独立，不共享可变状态
  - Router 内存结构仅由 event loop 单线程访问，天然无竞态
```

### 与 ZMQ socket 的配合

#### ROUTER socket 配置

```
ROUTER:
  RCVHWM = 10000      # 接收高水位，配合背压
  SNDHWM = 10000      # 发送高水位
  IMMEDIATE = 1        # 立即发送，不等 HWM
  ROUTER_MANDATORY = 0 # 不强制路由，安全丢弃

XPUB:
  SNDHWM = 10000      # 发送高水位
  IMMEDIATE = 1        # 立即发送
```

#### 为什么不用 ZMQ 内置的 IO 线程

```
ZMQ 内置 IO 线程处理网络收发，应用层只处理消息体。
asyncio 封装后，协程调度取代了传统回调。
应用层的并发模型（asyncio + 信号量）和 ZMQ 的 IO 线程是正交的，
不需要也不需要能干预 ZMQ IO 线程。
```

### 对象池：Context 复用

#### 问题

消息主循环每处理一条消息，都需要创建临时上下文对象（用于 pipeline 传递的 dict、临时的 Message 包装等）。在 50000 msg/s 下每秒产生 5 万+ 个短生命周期对象，触发频繁 GC。

#### 设计

```python
class MessageContextPool:
    """消息上下文对象池，预分配 + 复用"""

    def __init__(self, size: int = 1000):
        self._pool: list[dict] = [{} for _ in range(size)]
        self._available: int = size

    def acquire(self) -> dict:
        """获取一个干净的上下文 dict"""
        if self._available > 0:
            self._available -= 1
            return self._pool[self._available]
        return {}  # 池耗尽时回退到普通分配

    def release(self, ctx: dict):
        """归还上下文，清空后放回池"""
        ctx.clear()
        if self._available < len(self._pool):
            self._pool[self._available] = ctx
            self._available += 1
```

#### 集成到消息循环

```
_process_topic_batch(topic, msgs):
  async with sem(并发上限=100):
    for msg in msgs:
      ctx = ctx_pool.acquire()         ← 从池获取
      try:
        ctx["identity"] = msg.identity
        ctx["topic"] = msg.topic
        ctx["meta"] = msg.meta
        ctx["payload"] = msg.payload
        pipeline.execute(ctx)
        handler.dispatch(ctx)
      except ...:
        ...
      finally:
        ctx_pool.release(ctx)          ← 归还给池
```

#### 收益

| 指标 | 无池 | 有池 (size=1000) |
|------|------|-------------------|
| dict 分配/秒 (50000 msg/s) | 50000 | ~0 (池命中) |
| GC 第 0 代回收频率 | 高 | 极低 |
| P99 延迟抖动 | 受 GC 影响 | 平稳 |

#### 配置

池大小由 `object_pool_size` 配置项控制，默认 1000。

### 设计优势

| 优势 | 说明 |
|---|---|
| 低负载低延迟 | batch_size=1，每条消息即时处理，零额外等待 |
| 高负载高吞吐 | 批量收+派发，减少协程调度开销 |
| 自动背压 | 信号量阈值触发 → ZMQ HWM → 传导至 Producer |
| 渐进降级 | 不会在某条消息上突然阻塞所有流量 |

| 无锁无额外组件 | 纯 asyncio 协程 + 信号量，零锁依赖 |

| 跨 Topic 并发 | 不同 topic 独立派发 task，不互相阻塞 |
| 错误隔离 | 单条消息异常不影响其他消息 |

## 二、Router 内部结构

### 定位

引擎层的纯内存消息路由，三个核心子组件：

```
Router
├── TopicRegistry        → 注册 topic，支持通配符匹配
├── SubscriptionManager  → 管理订阅关系（双向索引）
├── ConnectionManager    → identity ↔ user 映射
└── MessageBuffer        → 每个 topic 内存环形缓冲区 (最近 1000 条)
```

全部纯内存，不持久化，不落库。

---

### 一、TopicRegistry

#### 职责

- SUB 消息收到新 topic → 自动注册
- PUB 消息到达 → 查找 topic 存在性 + 解析通配符订阅

#### 数据结构

```
_exact: Dict[str, TopicInfo]              # 精确 topic → 元信息
_wildcards: Dict[str, List[TopicInfo]]    # namespace → 通配符 topic 列表
_wildcard_cache: LRU(key → match_result)  # 通配符匹配结果缓存 (TTL=5s)

TopicInfo:
  full_name: str          "team-a.mkt.sh.600000"
  namespace: str          "team-a"          ← topic 第一段
  topic_path: str         "mkt.sh.600000"
  is_wildcard: bool
  subscriber_count: int   活跃订阅数（统计用途）
  created_at: float
```

#### 操作复杂度

| 操作 | 复杂度 | 说明 |
|---|---|---|
| register(topic) | O(1) | dict 插入 |
| exists(topic) | O(1) | dict 查找 |
| resolve_wildcards(pattern) | O(1) 缓存命中 / O(N_w) 未命中 | N_w = 该 namespace 的通配符数 |

#### 通配符匹配优化

通配符 topic 数量通常很少（几十个 group 级 pattern），直接遍历即可，配合缓存：

```
resolve_wildcards("team-a.mkt.sh.600000"):
  1. 检查缓存 → 命中直接返回
  2. 遍历 team-a 的通配符列表:
     "team-a.mkt.*"     → 匹配 ✓
     "team-a.signal.*"  → 不匹配
     "*.mkt.*"          → 匹配 ✓
  3. 缓存结果 (TTL=5s)
  4. 返回匹配列表
```

**通配符规则**：

| 符号 | 含义 | 示例 | 匹配 | 不匹配 |
|---|---|---|---|---|
| `*` | 单层匹配 | `a.*.c` | `a.b.c` | `a.b.x.c`, `a.c` |
| `>` | 多层匹配 | `a.>.c` | `a.b.c`, `a.b.x.c` | `b.c`, `a.c` |

### 二、SubscriptionManager

#### 职责

- SUB → 建立 identity → topic 映射
- UNSUB → 删除映射
- PUB → 获取 topic 的所有订阅者
- 连接断开 → 清理该 identity 的所有订阅

#### 数据结构（双向索引）

```
正向: _topic_subscribers: Dict[str, Set[bytes]]
  "team-a.mkt.sh.600000" → {identity_1, identity_2, identity_3}

反向: _identity_subscriptions: Dict[bytes, Set[str]]
  identity_1 → {"team-a.mkt.sh.600000", "team-a.mkt.sz.000333"}
```

**为什么需要双向**：断开连接时，如果只有 topic→subscribers，需要遍历所有 topic 才能找到该 identity 的订阅。双向索引两个方向都是 O(1)。

#### 操作复杂度

| 操作 | 复杂度 | 实现 |
|---|---|---|
| subscribe(identity, topic) | O(1) | 两个 dict 的 set add |
| unsubscribe(identity, topic) | O(1) | 两个 dict 的 set discard |
| get_subscribers(topic) | O(1) | dict get → 返回引用（调用方只读） |
| get_subscriptions(identity) | O(1) | dict get → 返回引用 |
| remove_identity(identity) | O(N_sub) | N_sub = 该 identity 的订阅数，通常很小 |

#### 通配符订阅处理（关键优化）

**策略**：订阅时展开，广播时直接查。

```
SUB "team-a.mkt.*"（通配符）:
  1. TopicRegistry.register("team-a.mkt.*") → 注册为通配符
  2. 解析已注册的精确 topic:
     "team-a.mkt.sh.600000" → 匹配 → subscribe(identity, exact_topic)
     "team-a.mkt.sz.000333" → 匹配 → subscribe(identity, exact_topic)
  3. 记录通配符本身: subscribe(identity, "team-a.mkt.*")

UNSUB "team-a.mkt.*":
  1. 解析匹配的精确 topic → 逐一 unsubscribe
  2. 如果通配符没有其他订阅者 → 从 TopicRegistry 移除
```

**为什么"订阅时展开"优于"广播时展开"**：

| 方式 | PUB 每条消息开销 | 说明 |
|---|---|---|
| 广播时展开（旧） | O(N_w) 通配符匹配 | 每条消息都做一次匹配 |
| 订阅时展开（新） | O(1) dict 查找 | 不用再匹配通配符 |

#### PUB 广播流程（最终效果）

```
PUB topic="team-a.mkt.sh.600000":
  subscribers = get_subscribers(topic)  → {id_1, id_2, ...}  O(1)
  for identity in subscribers:
    transport.broadcast(identity, frames[2:4])  ← 零拷贝
```

### 三、ConnectionManager

#### 职责

- AUTH 成功 → identity ↔ user 映射
- 每条消息处理 → 从 identity 获取 user
- 连接断开 → 移除映射 + 清理订阅

#### 数据结构

```
_identity_user: Dict[bytes, AuthUser]
  identity_1 → {user_id: 1, role: "user", groups: [...], namespace: "team-a"}

_user_identities: Dict[int, Set[bytes]]
  user_id_1 → {identity_1, identity_2}     ← 支持多连接
```

#### AuthUser

```
user_id: int
role: "admin" | "user"
groups: List[str]          ← 权限组名称列表
api_key: str
namespace: str             ← home_namespace，仅作客户端默认前缀提示
```

#### 操作复杂度

| 操作 | 复杂度 | 说明 |
|---|---|---|
| register(identity, user) | O(1) | dict 插入 |
| unregister(identity) | O(1) | dict 删除 + set discard |
| get_user(identity) | O(1) | dict get |
| get_identities(user_id) | O(1) | dict get → 返回引用 |
| kick_user(user_id) | O(N_conn) | N_conn = 用户连接数 |

### 四、MessageBuffer（内存环形缓冲区）

#### 定位

不持久化到磁盘，每个 topic 在内存中保留最近 N 条消息。用于：
- 新订阅者上线后快速回放历史数据（最近 1000 条）
- 客户端短暂断线重连后补数据（跳过持久化的复杂性）

#### 设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 持久化 | ❌ 不落盘 | 追求低延迟，持久化会引入 IO 瓶颈 |
| 保留条数 | 1000 条/topic | 覆盖大多数场景（以 tick 级行情计，约 16 分钟数据） |
| 淘汰策略 | 环形覆盖 | 始终保留最新 N 条，旧数据自动覆盖 |
| 数据结构 | `collections.deque` | O(1) append + O(1) popleft |
| 内存预估 | 1KB/条 × 1000 × 1000 topic ≈ 1GB | 单机 1000 topic 仅 1MB，实际远低于此 |

#### 数据结构

```python
class MessageBuffer:
    """每个 topic 的内存环形缓冲区"""

    MAX_SIZE: int = 1000

    # per_topic: Dict[str, Deque[BufferedMessage]]
    # key = topic, value = 环形队列（最新在右侧）
    _buffers: Dict[str, deque]

    _seq_counter: Dict[str, int]   # topic → 下一个序列号
```

```python
@dataclass(slots=True)
class BufferedMessage:
    topic: str
    seq: int                # topic 内单调递增消息序号
    record_count: int       # 本条消息携带的数据行数（≥1）
    meta: bytes             # Frame 3 的 2 字节（含 ser_fmt + comp flag + has_count）
    payload: bytes          # Frame 5 的序列化+压缩后的 payload
    timestamp: float        # 服务端 接收时间
```

`slots=True`（Python 3.10+）消除了每个实例的 `__dict__`（约 56 字节/对象），在满载 100 万条消息时节省约 **50MB** 内存并减少 GC 扫描开销。

#### 操作接口

```python
class MessageBuffer:
    def append(self, topic: str, meta: bytes, record_count: int,
               payload: bytes) -> BufferedMessage:
        """PUB 消息广播后调用，追加到缓冲区"""
        seq = self._seq_counter.get(topic, 0) + 1
        self._seq_counter[topic] = seq

        msg = BufferedMessage(topic=topic, seq=seq,
                              record_count=record_count,
                              meta=meta, payload=payload,
                              timestamp=time.time())

        if topic not in self._buffers:
            self._buffers[topic] = deque(maxlen=self.MAX_SIZE)
        self._buffers[topic].append(msg)
        return msg

    def replay(self, topic: str, from_seq: int = 0,
               limit: int = 100) -> List[BufferedMessage]:
        """从指定序列号开始回放消息"""
        if topic not in self._buffers:
            return []
        msgs = [m for m in self._buffers[topic] if m.seq >= from_seq]
        return msgs[:limit]

    def latest_seq(self, topic: str) -> int:
        """获取 topic 最新序列号"""
        return self._seq_counter.get(topic, 0)

    def remove_topic(self, topic: str):
        """topic 无订阅者时清理缓冲区"""
        self._buffers.pop(topic, None)
        self._seq_counter.pop(topic, None)
```

#### 与 PUB 路径集成

```
PUB 消息处理完成后:
```

## 三、拦截器链

### 定位

引擎层的可插拔消息处理链。每条消息在到达 Handler 之前，必须经过拦截器链校验。

---

### 抽象接口

#### Interceptor

```
async intercept(context: PipelineContext, next: Callable) -> None

规则:
  - 必须在方法内调用 next() 才能继续链
  - 不调用 next() = 短路终止（静默丢弃消息）
  - 抛出异常 = 错误终止（被 Engine 的 try/catch 捕获，返回 ERROR）
```

#### PipelineContext

```
identity: bytes          ← ZMQ identity（每消息唯一）
msg_type: int            ← msg_type 枚举值
topic: str | None        ← 从 Frame 2 获取
meta: (int, int)         ← 从 Frame 3 获取 (msg_type, flags)
payload: bytes           ← 从 Frame 5 获取，不解码
user: AuthUser | None    ← AuthInterceptor 注入，PermInterceptor 读取
timestamp: float         ← 消息到达时间戳
```

直接属性访问，无需通用 KV 存储。目前 3 个拦截器仅需 `context.user` 一个中间传递字段。

#### InterceptorChain

```
async execute(context: PipelineContext)

内部实现: 递归调用链中每个拦截器的 intercept() 方法
```

---

### 默认链顺序

```
AuthInterceptor → PermissionInterceptor → MonitorInterceptor
```

#### 为什么这个顺序

| 顺序 | 理由 |
|---|---|
| Auth 先 | 非法消息尽早拒绝，减少后续计算 |
| Perm 其次 | 认证通过后才校验权限，减少缓存穿透 |
| Monitor 最后 | 记录所有通过链（或被拒绝）的消息，无论成功失败 |

---

### 各拦截器设计

#### AuthInterceptor

```
触发条件: 除 AUTH 消息外的所有消息

流程:
  1. msg_type == AUTH → 跳过，直接 next()
  2. user = connection_manager.get_user(identity)
  3. user == None → 抛出 AuthError("not authenticated")
  4. context.user = user
  5. await next()           ← 继续链

短路: user == None → 抛出异常（不调用 next()）
```

#### PermissionInterceptor

```
触发条件: PUB / SUB / UNSUB / QUERY

流程:
  1. user.role == "admin" → 跳过，直接 next()
  2. action = msg_type_to_action(msg_type)
     PUB→"pub", SUB→"sub", UNSUB→"sub", QUERY→"query"
  3. 查用户展开权限缓存: user_perms:{user.id}
     命中 → 匹配 topic_pattern 列表
     未命中 → 查 DB (UserGroup + GroupPermission) → 回写缓存
  4. topic 匹配任一 pattern + action 匹配 → 通过 → next()
  5. 不匹配 → 抛出 PermissionError("denied")

短路: 权限不足 → 抛出异常
```

#### MonitorInterceptor

```
触发条件: 所有消息

流程:
  1. start_ts = time.now()
  2. try:
       await next()
     finally:
       elapsed = time.now() - start_ts
       metrics.inc_message(topic, len(payload), elapsed_ms)
  
  如果 next() 抛出异常:
    finally 中记录: metrics.inc_error(exception.type)

设计要点:
  - next() 在 try/finally 中，确保异常时也能记录
  - 不修改 context，不影响其他拦截器
  - 不抛出异常（纯旁路记录）
```

---

### 扩展点

#### 添加限流拦截器（示例）

```
class RateLimitInterceptor:
    def __init__(self, rate_limiter):
        self._limiter = rate_limiter

    async def intercept(self, context, next):
        if not self._limiter.allow(context.user.id):
            raise RateLimitError("rate limit exceeded")
        await next()

# 插入链
chain = InterceptorChain([
    AuthInterceptor(),
    PermissionInterceptor(),
    RateLimitInterceptor(),      ← 在权限之后、监控之前
    MonitorInterceptor()
])
```

#### 可添加的拦截器类型

| 拦截器 | 功能 | 位置建议 |
|---|---|---|
| RateLimitInterceptor | 用户级别 QPS 限流 | Perm 之后 |
| AuditInterceptor | 审计日志 | Monitor 之后 |
| FeatureFlagInterceptor | 灰度开关 | Auth 之后 |
| MessageValidator | 消息内容校验（schema） | Auth 之后 |

---

### 与 Engine 的集成

```
Engine._process(msg):

  context = PipelineContext(identity, msg_type, topic, meta, payload)

  try:
    await pipeline.execute(context)       ← 拦截器链

    response = await handler.dispatch(context)   ← 消息处理

    if response:
      transport.send(identity, encode(response))

  except AuthError:
    transport.send(identity, ERROR(1001, "auth required"))
  except PermissionError:
    transport.send(identity, ERROR(2001, "permission denied"))
  except RateLimitError:
    transport.send(identity, ERROR(2003, "rate limited"))
  except Exception:
    log.error()
    transport.send(identity, ERROR(9001, "internal error"))
```

**关键**：拦截器链的异常被 Engine 统一捕获，不区分是哪个拦截器抛出的。

## 四、过载保护策略

### 定位

防止高负载时系统内存爆炸、消息无限堆积。采用**双缓冲 + 优先级丢弃**方案。

---

### 三层防线

```
Layer 1: 信号量阈值 (应用层)
  pending_tasks > max_concurrency * 0.8 → 暂停 recv，等水位下降

Layer 2: ZMQ HWM (内核层)
  RCVHWM=10000, SNDHWM=10000 → socket 缓冲区满自动背压

Layer 3: 双缓冲 + 优先级丢弃 (应用层，最后一道防线)
  控制消息和数据消息分池存储，数据消息满了直接丢
```

---

### 核心设计：双缓冲

#### 数据结构

```
data_buffer: Deque[Message]   容量=9000   ← 行情数据
ctrl_buffer: Deque[Message]   容量=1000   ← 控制消息

为什么双缓冲:
  - 控制消息(AUTH/SUB/QUERY)量小但重要，不能被行情数据挤掉
  - 数据消息(PUB)量大但可丢弃，不影响系统可用性
  - 分离后控制路径永不饿死
```

#### 消息分类

```
控制消息 (进 ctrl_buffer):
  AUTH  → meta[0]=0x01
  SUB   → meta[0]=0x03
  UNSUB → meta[0]=0x04
  QUERY → meta[0]=0x05
  PING  → meta[0]=0x06

数据消息 (进 data_buffer):
  PUB   → meta[0]=0x02

STATUS → 进 data_buffer（低优先级）
ERROR  → 不经过缓冲区（服务端主动发出）
```

#### 分流逻辑

```
Engine.recv() 收到消息:
  msg_type = meta[0]

  if msg_type in (AUTH, SUB, UNSUB, QUERY, PING):
    if len(ctrl_buffer) < ctrl_buffer_size:
      ctrl_buffer.append(msg)
    else:
      ctrl_buffer.popleft()    ← 丢弃最旧的控制消息
      ctrl_buffer.append(msg)

  else:  # PUB / STATUS
    if len(data_buffer) < data_buffer_size:
      data_buffer.append(msg)
    else:
      metrics.dropped_total++   ← 数据消息满了直接丢
      # 不做插入，保持 FIFO
```

---

### 消费逻辑

```
Engine 消息循环:

while running:
  # 先消费控制消息（保证响应）
  while ctrl_buffer 非空 且 信号量有空位:
    msg = ctrl_buffer.popleft()
    dispatch(msg)

  # 再消费数据消息
  while data_buffer 非空 且 信号量有空位:
    msg = data_buffer.popleft()
    dispatch(msg)

  # 缓冲区空 → recv 新消息
```

**保证**：控制消息不会因为行情爆发而被饿死。

---

### 对象池：Message 复用

#### 问题

高负载时双缓冲频繁 append/popleft，每条消息创建一个新的 `Message` 包装对象。50000 msg/s 下每秒产生 5 万个临时对象，增加 GC 压力。

#### 设计

双缓冲中的 `Message` 使用对象池复用，避免频繁分配/释放：

```python
class MessagePool:
    """Message 对象池，预分配 + 复用"""

    def __init__(self, size: int = 4096):
        self._pool: list[Message] = [Message.__new__(Message) for _ in range(size)]
        self._available: int = size

    def acquire(self) -> Message:
        """从池获取一个干净的 Message"""
        if self._available > 0:
            self._available -= 1
            return self._pool[self._available]
        return Message.__new__(Message)  # 池耗尽回退

    def release(self, msg: Message):
        """归还 Message，重置字段后放回池"""
        msg.reset()  # 清空 identity/topic/meta/payload 等字段
        if self._available < len(self._pool):
            self._pool[self._available] = msg
            self._available += 1
```

#### 集成到分流逻辑

```
Engine.recv() 收到消息:
  msg = msg_pool.acquire()          ← 从池获取
  msg.identity = frames[0]
  msg.topic = frames[2]
  msg.meta = frames[3]
  ...

  if msg_type in (AUTH, SUB, UNSUB, QUERY, PING):
    ctrl_buffer.append(msg)
  else:
    data_buffer.append(msg)

消费完成后:
  msg = data_buffer.popleft()
  dispatch(msg)
  msg_pool.release(msg)             ← 归还给池
```

#### 收益

| 指标 | 无池 | 有池 (size=4096) |
|------|------|-------------------|
| Message 分配/秒 | ~50000 | ~0 (池命中) |
| GC 暂停频率 | 高 | 极低 |
| 内存碎片 | 累积 | 几乎无 |

#### 配置

池大小由 `object_pool_size` 配置项控制。

---

### 配置参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| max_concurrency | 100 | 信号量上限 |
| backpressure_threshold | 0.8 | 信号量使用率 80% → 暂停 recv |
| zmq_rcvhwm | 10000 | ZMQ 接收高水位 |
| zmq_sndhwm | 10000 | ZMQ 发送高水位 |
| data_buffer_size | 9000 | 数据缓冲区容量 (HWM * 0.9) |
| ctrl_buffer_size | 1000 | 控制缓冲区容量 (HWM * 0.1) |
| object_pool_size | 4096 | Message 对象池预分配数量 |

---

### 背压传导链路

```
Producer 发送过快
    ↓
Server 消息堆积 → data_buffer 满
    ↓
新到达 PUB 被丢弃，dropped_total++
    ↓
如果持续堆积 → 信号量也满
    ↓
Server 暂停 recv
    ↓
ZMQ RCVHWM 触发 → Producer send 阻塞或失败
    ↓
Producer 感知 → 降低速率或告警
```

---

### 监控指标

| 指标 | 说明 |
|---|---|
| `backpressure` | 当前是否处于背压 (bool) |
| `dropped_total` | 累计丢弃消息数 |
| `data_buffer_usage` | 数据缓冲区使用百分比 |
| `ctrl_buffer_usage` | 控制缓冲区使用百分比 |
| `pending_tasks` | 当前待处理 task 数 |

---

### 为什么不用复杂的优先级丢弃算法

复杂的算法（如遍历缓冲区找更低优先级的消息来替换）有两个问题：

1. **O(N) 遍历**：大量消息时遍历开销不可忽略
2. **公平性问题**：后到的"高优先级"消息覆盖先到的"低优先级"消息，旧消息永远得不到处理

双缓冲 + 直接丢弃是 **O(1)** 的，且语义清晰：

| 场景 | 行为 | 原因 |
|---|---|---|
| 控制消息满 | 丢弃最旧的，插入新的 | 控制消息不丢了就行 |
| 数据消息满 | 直接丢弃新消息 | FIFO 保证已有数据按顺序处理，丢弃新消息让 Producer 感知背压 |

---

### 边界情况

#### 控制缓冲区满 → 控制消息被丢弃

```
这意味着:
  - AUTH 丢失: 客户端没有收到认证响应，超时后重连
  - SUB 丢失: 客户端超时后重试
  - PING 丢失: 心跳探测失败，但 ZMQ 连接仍存活

不会导致:
  - 数据路径被控制消息阻塞 (有独立缓冲区)
  - 系统不可用 (控制消息量极小，几乎不可能满)
```

#### ZMQ HWM 触发 → Producer 阻塞

```
ZMQ 内核层背压:
  Producer send() → 阻塞（ZMQ socket send 默认阻塞）
  → Producer 发现发送变慢 → 自行降速
  → 这是一道安全防线，不为应用层处理
```

---

### 与消息主循环的关系

双缓冲是 Engine 中 recv → dispatch 中间的"蓄水池"：

```
recv → 分流到双缓冲 → 消费循环 → 信号量限流 → dispatch
```

缓冲的存在让 recv 持续排空 ZMQ socket（快速释放内核缓冲区），消费端按自己的节奏处理，两者解耦。
